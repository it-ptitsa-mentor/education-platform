Сравнение

Стратегия Где рендерится Когда Свежесть данных SEO Нужен сервер
SSGStatic Site Generation Сервер, заранее При сборке (build) Заморожена до пересборки отличное Нет — раздача с CDN
ISRIncremental Static Regeneration Сервер, фоном Сборка + ре-валидация по таймеру Обновляется раз в N секунд отличное Да (инфра Next)
SSRServer-Side Rendering Сервер На каждый запрос Всегда свежая отличное Да — живой Node
CSRClient-Side Rendering Браузер В рантайме, после загрузки JS Любая — клиент грузит сам слабое Нет — отдаётся статика

После SSR/SSG сервер прислал готовый HTML, но кнопки ещё «мёртвые» — браузер навешивает обработчики поверх разметки. Этот шаг называется гидрация (hydration).

Что ещё спросят рядом

RSC

React Server Components

Компоненты, которые выполняются только на сервере и вообще не попадают в JS-бандл браузера. Можно ходить в БД и держать секреты прямо внутри.

Не путать с SSR: SSR рендерит HTML, но всё равно шлёт JS компонентов для гидрации. RSC — не шлёт совсем.
STREAMING SSR

Потоковый SSR

Сервер отдаёт HTML не целиком, а кусками по мере готовности через <Suspense>.

Быстрые части экрана показываются сразу, медленные (ждущие тяжёлый запрос) дорисовываются потоком — вся страница больше не ждёт самый долгий запрос.

Как выбирать

Чем данные более общие и стабильные — тем левее к SSG. Чем более персональные и живые — тем правее к SSR / CSR.

SSG Один контент для всех, меняется редко. Блог, лендинг, документация.
ISR Один для всех, но обновляется периодически. Каталог товаров, цены раз в час.
SSR У каждого своё / меняется на каждый запрос. Лента, профиль, корзина.
CSR Насыщенный интерактив после загрузки, SEO не важно. Админки, дашборды за логином.

Это не «или-или» на весь сайт: Next / Nuxt позволяют выбирать стратегию на каждую страницу. Лендинг — SSG, каталог — ISR, личный кабинет — CSR/SSR.
