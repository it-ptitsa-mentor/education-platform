# Инверсия зависимостей

Не всегда результат функции совпадает с побочным эффектом, как в прошлом уроке. Иногда побочный эффект — лишь дополнительное действие, которое мешает проверить основную логику.

Возьмём функцию регистрации пользователя: она создаёт запись в базе и отправляет приветственное письмо:

```
const params = {
  email: 'lala@example.com',
  password: 'qwerty',
}
registerUser(params)
```

Функция делает многое, но нас интересует главное — корректная регистрация. Обычно это запись о новом пользователе в БД. Именно её и проверяют: есть ли новая запись с верными полями. Возвращаемое значение функции здесь мало помогает.

Как правило, базу в тестах не прячут. Во фреймворках она доступна в тестовой среде и работает как в проде. Идемпотентность достигают транзакциями: перед тестом транзакция открывается, после — откатывается. Каждый тест стартует в одинаковом окружении, независимо от того, что он меняет:

```
// Гипотетический пример
const ctx = /* connect to db */;
beforeEach(() => ctx.beginTransaction());

test('registerUser', () => {
  // Внутри идет добавление данных в базу
  const id = registerUser({ name: 'Mike' });
  const user = User.find(id);
  expect(user).toHaveProperty('name', 'Mike');
})

// За счет отката база возвращается в исходное состояние
afterEach(() => ctx.rollbackTransaction());
```

С отправкой писем сложнее: в тестах её выполнять нельзя. Как добиться нужного поведения? Примерно так может выглядеть регистрация:

```
import sendEmail from './emailSender.js'
const registerUser = (params) => {
  const user = new User(params)
  if (user.save()) {
    sendEmail('registration', { user })
    return true
  }
  return false
}
```

Есть несколько способов отключить отправку в тестах. Простейший — переменная окружения, указывающая среду выполнения:

```
// Выполняем этот код только если мы не в тестовой среде
if (process.env.NODE_ENV !== 'test') {
  sendEmail('registration', { user })
}
```

Подход прост, но считается плохой практикой: нарушается абстракция — код узнаёт, где он запущен. Со временем таких проверок накапливается больше, и код «загрязняется». К тому же, если нужно убедиться, что письмо ушло с правильными данными, такой вариант не поможет.

Следующий вариант — режим тестирования внутри библиотеки. На этапе инициализации тестов можно сделать так:

```
// setup.js в jest
import sendEmail from './emailSender.js'

// У этого подхода много разновидностей, начиная от установки флага,
// заканчивая заменой функций в прототипе.
sendEmail.test = true
```

После этого везде, где вызывается `sendEmail()`, реальной отправки не будет:

```
// Ничего не происходит
sendEmail('registration', { user })
// В отличие от первого варианта, прикладной код ни о чем не догадывается
```

Решение распространённое; как включить тестовый режим, обычно описано в документации библиотеки.

Если библиотека не поддерживает тестовый режим, остаётся универсальный приём — инверсия зависимостей. Код можно перестроить так, чтобы `sendEmail()` вызывалась не напрямую, а передавалась параметром:

```
import sendEmail from './emailSender.js'

// Ставим значение по умолчанию, чтобы не пришлось постоянно указывать функцию
const registerUser = (params, send = sendEmail) => {
  const user = new User(params)
  if (user.save()) {
    send('registration', { user })
    return true
  }
  return false
}
```

Сначала создают функцию-замену, имитирующую отправку без реальной доставки. Внутри можно что-то логировать или оставить тело пустым:

```
const fakeSendEmail = (...args) => {
  // Тут выполняем какие-то действия
  // Или вообще ничего не делаем
}
```

В тесте подставляют эту функцию при регистрации:

```
test('registerUser', () => {
  const id = registerUser({ name: 'Mike' }, fakeSendEmail)
  const user = User.find(id)
  expect(user).toHaveProperty('name', 'Mike')
})
```

Вызов внутри `registerUser()` сработает, но письмо не уйдёт.

Способ сложнее, особенно если функция глубоко в стеке вызовов: зависимости придётся протаскивать по всей цепочке. Их может быть много — чем больше инверсий, тем тяжелее код. Зато гибкость выше.

Плюсы: ни библиотека, ни приложение не знают о тестах. Можно задавать поведение под конкретный сценарий. В ряде экосистем (PHP, Java, C#) инверсия зависимостей встроена в процесс сборки.
