# Тестирование ошибок

Базовые тесты обычно покрывают успешные сценарии. Но код иногда должен падать с исключением — и это тоже нужно проверять. Речь о ситуациях, когда выбрасывается ошибка. В чём подвох? Посмотрите на тест:

```
test('boom!', () => {
  try {
    functionWithException(0)
  }
  catch (e) {
    expect(e).not.toBeNull()
  }
})
```

Тест пытается убедиться, что `functionWithException()` бросает исключение при аргументе 0. Действительно ли он это проверяет?

Нет. Если исключения не будет, выполнение не зайдёт в `catch`, и тест всё равно пройдёт.

В документации Jest есть обходной путь — задать число утверждений, которые обязаны выполниться:

```
test('boom!', () => {
  // Количество утверждений, которые должны быть запущены в этом тесте
  expect.assertions(1)

  try {
    functionWithException(0)
  }
  catch (e) {
    expect(e).not.toBeNull()
  }
})
```

Такой приём опасен. Тест становится хрупким и привязанным к форме записи. Добавите ещё одно утверждение — придётся менять счётчик. Постоянно следить за числом утруднительно. Лучше не использовать этот подход: чем больше скрытой связности, тем сложнее читать код и тем выше риск ошибки.

Правильный вариант в Jest — матчер, который сам ловит исключение и проверяет, что оно возникло:

```
test('boom!', () => {
  expect(() => {
    functionWithException(0)
  }).toThrow()
})
```

Матчер принимает функцию и вызывает её внутри — поэтому он видит выброс. Нет неявного состояния и лишних проверок; код делает ровно то, что нужно. Теоретически в одном тесте можно проверить несколько разных исключений — с `try/catch` это заметно сложнее.

Иногда важно не только поймать ошибку, но и убедиться, что это та самая. В `toThrow()` можно передать подстроку, которая должна встретиться в сообщении:

```
test('boom!', () => {
  expect(() => {
    functionWithException(0)
  }).toThrow('divide by zero')
})
```
