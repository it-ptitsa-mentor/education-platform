# Моки

В тестировании широко применяют мокинг. Технически он близок к стабингу, из‑за чего их часто смешивают. Цели и ситуации применения всё же разные — разберёмся.

Раньше побочные эффекты мы рассматривали как помеху проверке логики. Их изолировали стабами или отключением в тестовой среде, после чего можно было спокойно проверять результат функции.

Иногда нужно другое: не результат, а факт нужного действия — например, что ушёл корректный HTTP-запрос с верными параметрами. Здесь нужны моки. Они проверяют, *как* выполняется код.

## HTTP

```
import nock from 'nock'
import { getPrivateForkNames } from '../src.js'

// Предотвращение случайных запросов
nock.disableNetConnect()

test('getPrivateForkNames', async () => {
  // Полное название домена
  const scope = nock('https://api.github.com')
    // Полный путь
    .get('/orgs/yandex/repos/?private=true')
    .reply(200, [{ fork: true, name: 'one' }, { fork: false, name: 'two' }])

  await getPrivateForkNames('yandex')
  // Метод `scope.isDone()` возвращает `true` только тогда,
  // когда соответствующий запрос был выполнен внутри `getPrivateForkNames`
  expect(scope.isDone()).toBe(true)
})
```

Это и есть мокинг. Мок убеждается, что код отработал определённым образом: вызов функции, HTTP-запрос и т.п. Важно и то, *что* произошло, и *как* — с какими аргументами.

В примере с GitHub такая проверка мало что даёт сама по себе: факт вызова ещё не гарантирует корректность. Когда моки действительно полезны?

Представьте, что вы пишете библиотеку *@octokit/rest* для GitHub API. Её суть — правильные запросы с правильными параметрами. Тогда проверка точных URL обязательна: иначе нельзя быть уверенным в поведении клиента.

Отсюда отличие мока от стаба. Стаб убирает побочный эффект, чтобы не мешал проверке результата (например, возврата данных). Мок смотрит на внутреннее поведение. Технически мок и стаб создают похоже, но на мок вешают ожидания вызовов — отсюда путаница в терминах. Для себя важно понимать, о чём речь: от этого зависит фокус теста.

## Функции

Моки часто применяют к функциям (методам). Можно проверить:

- что функция вызывалась;

- сколько раз;

- с какими аргументами;

- сколько аргументов передано;

- что вернула функция.

Допустим, тестируем `forEach` — она вызывает колбэк для каждого элемента:

```
[1, 2, 3].forEach(v => console.log(v)) // или проще [1, 2, 3].forEach(console.log)
```

Функция ничего не возвращает, напрямую её не проверить. Попробуем моки — убедимся, что колбэк вызывается с нужными значениями.

*Мы изучаем Jest, поэтому используем встроенный механизм. В других фреймворках — свои средства. Есть и отдельные библиотеки для моков и стабов.*

```
test('forEach', () => {
  // Моки функций в Jest создаются с помощью функции jest.fn
  // Она возвращает функцию, которая запоминает все свои вызовы и переданные аргументы
  // Потом это используется для проверок
  const callback = jest.fn();

  [1, 2, 3].forEach(callback)

  // Теперь проверяем, что она была вызвана с правильными аргументами нужное количество раз
  expect(callback.mock.calls).toHaveLength(3)
  // Первый аргумент первого вызова
  expect(callback.mock.calls[0][0]).toBe(1)
  // Первый аргумент второго вызова
  expect(callback.mock.calls[1][0]).toBe(2)
  // Первый аргумент третьего вызова
  expect(callback.mock.calls[2][0]).toBe(3)
})
```

Моки подтвердили три вызова и передачу элементов по очереди. Тест вроде бы проверяет `forEach()`. Но можно проще — без мока и без привязки к внутренностям — через замыкание:

```
test('forEach', () => {
  const result = []
  const numbers = [1, 2, 3]
  numbers.forEach(x => result.push(x))
  expect(result).toEqual(numbers)
})
```

## Объекты

В Jest мокируют и объекты — через `jest.spyOn()`, по аналогии с `jest.fn()`. Функция принимает объект и имя метода и отслеживает вызовы. Пример с `console.log()`:

```
test('logging something', () => {
  const spy = jest.spyOn(console, 'log')

  console.log(12)

  expect(spy).toHaveBeenCalled() // => true, т.к. метод log был вызван
  expect(spy).toHaveBeenCalledTimes(1) // => true, т.к. метод был вызван 1 раз
  expect(spy).toHaveBeenCalledWith(12) // true, т.к. log был вызван с аргументом 12
  expect(spy.mock.calls[0][0]).toBe(12) // проверка, идентичная предыдущей
})
```

Jest также позволяет задать свою логику сбора вызовов через `mockImplementation(fn)`. Колбэк выполняется после каждого вызова отслеживаемой функции. Соберём аргументы каждого `console.log()` в массив:

```
test('logging something', () => {
  const logCalls = []
  // Здесь функция внутри mockImplementation принимает на вход аргумент(ы),
  // с которым был вызван console.log, и сохраняет его в заранее созданный массив
  const spy = jest.spyOn(console, 'log').mockImplementation((...args) => logCalls.push(args))

  console.log('one')
  console.log('two')

  expect(logCalls.join(' ')).toBe('one two')
})
```

## Преимущества и недостатки

Моки нужны в отдельных сценариях, но в большинстве случаев их лучше избегать. Мок слишком много знает о реализации. Тест с моками превращается из чёрного ящика в белый. Повсеместный мокинг ведёт к двум проблемам:

- после рефакторинга приходится переписывать много тестов, даже если поведение верное — из‑за привязки к устройству кода;

- код может сломаться, а тесты пройдут, потому что проверяется не результат, а внутренняя структура.

Где можно использовать реальный код — используйте реальный. Где можно обойтись без моков — обходитесь. Избыточный мокинг обесценивает тесты и удорожает поддержку. Идеал — тестирование чёрным ящиком.
