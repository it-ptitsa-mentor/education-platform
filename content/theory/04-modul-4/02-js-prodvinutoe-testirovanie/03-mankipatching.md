# Манкипатчинг

Инверсия зависимостей подходит не всегда: при глубоких вызовах код усложняется — зависимость приходится протаскивать через промежуточные функции. Есть приём, который позволяет добраться до нужных вызовов и подменить их без инверсии.

Пример класса с сетевым запросом в методе:

```
import axios from 'axios'

class Api {
  // ...
  async makeRequest(url) {
    const { data } = await axios.get(url)
    this.data = data
  }
}
```

Прототипная модель JS позволяет менять поведение объектов без прямого доступа к экземплярам — достаточно заменить методы в прототипе. Все объекты с этим прототипом начнут использовать новую реализацию.

Подменим метод, чтобы тесты не ходили в сеть:

```
// Подменяем метод makeRequest так, чтобы он не делал сетевой запрос
// После выполнения этого кода Api меняет свое поведение не только
// в этом модуле, но и вообще по всей программе
Api.prototype.makeRequest = function () {
  console.log('no request')
  this.data = 'test data'
}

// Где-то в другом файле
// Так как объекты передаются по ссылке, то это тот же Api
// что и в коде выше
import Api from './api.js'

const client = new Api()

// Вызывается подмененный makeRequest
client.makeRequest(/* аргументы не важны, внутри они не используются */)
// => 'no request'
```

Если объект (например, функция-конструктор) используется напрямую, достаточно изменить свойство самого объекта:

```
Array.isArray('') // false

// Этот код может быть вызван в любом месте программы
Array.isArray = () => true

Array.isArray('') // true

// То же самое касается любого импортируемого объекта
import Api from './api.js'

// Теперь везде, где будет импортироваться Api, это будет измененный Api
Api.boom = () => console.log('Magic')

// В любом другом модуле
Api.boom() // => 'Magic'
```

Глобальная подмена свойств называется манкипатчинг ([monkey patching](https://ru.wikipedia.org/wiki/Monkey_patch)). В прикладном коде это плохая практика, в тестах — распространённый и удобный приём.

Известный пример в JS — библиотека [nock](https://github.com/nock/nock). Она перехватывает сетевые запросы модуля [http](https://nodejs.org/api/http.html) из стандартной библиотеки Node.js.

```
// Пример http-запроса с использованием модуля http
import http from 'http'

const options = {
  hostname: 'yandex.io',
  port: 443,
  path: '/my',
  method: 'GET',
}

// request — асинхронный метод
const req = http.request(options, (res) => {
  // Тут обрабатываем http-ответ
})
```

Nock [заменяет](https://github.com/nock/nock/blob/v13.0.0-beta.5/lib/common.js#L94) внутри *http* методы, которыми пользуются HTTP-клиенты.

```
// Как примерно выглядит подмена

import http from 'http'

// Сохраняем старый метод
// Это позволяет вернуть его потом на место
const overriddenRequest = http.request

http.request = (/* тут такие же параметры как и у исходного метода */) => {
  // здесь логика библиотеки nock

  // Возвращаем исходный метод!
  http.request = overriddenRequest
}
```

Использование:

```
import nock from 'nock'
import { getPrivateForkNames } from '../src.js'

test('getPrivateForkNames', async () => {
  nock(/api\.github\.com/) // это регулярное выражение чтобы не указывать полный адрес
    // get — для GET-запросов, post — для POST-запросов и так далее
    .get(/\/orgs\/yandex\/repos/)
    .reply(200, [{ fork: true, name: 'one' }, { fork: false, name: 'two' }])

  const names = await getPrivateForkNames('yandex')
  expect(names).toEqual(['one'])
})
```

Цепочка `nock(domain).get(url)` задаёт адрес перехватываемого запроса. Nock сравнивает все запросы и подменяет совпадающий. Домен и путь можно задать строкой или регулярным выражением.

`reply(code, body, headers)` описывает ответ. В простейшем случае достаточно кода; здесь нужны ещё данные — по ним проверяется `getPrivateForkNames()`.

Мы рассмотрели базовый сценарий; у Nock обширная документация и много вариантов — имеет смысл заглядывать туда за более удобными приёмами.

Плюсы и минусы:

Главный плюс — почти универсальность. Подходит к любому коду без правок продакшена. Программа может не подозревать, что её тестируют.

Минус — тест из чёрного ящика становится «прозрачным»: он знает устройство кода и зависит от деталей. Тесты хрупкие: рефакторинг без смены поведения может сломать проверки из‑за привязки к домену, путям и формату ответа.

В большинстве проектов это терпимо — Nock можно смело применять, не забывая про альтернативы.
