Интерактивные системы — браузер, терминал и другие — устроены похоже: после загрузки они ждут действий пользователя (клики, ввод, движение мыши, сочетания клавиш).

В коде каждое такое действие представлено **событием**. Частые примеры:

- `click`

- `submit`

- `keyup`

- `keydown`

- `focus`

- `contextmenu`

- `mouseover`

- `mousedown`

- `mouseup`

События детализированы: нажатие клавиши даёт отдельные фазы `keydown` и `keyup`. Раньше для «печатных» символов использовали `keypress` — сейчас чаще смотрят на `keydown`/`keyup` и свойства клавиши.

Событие всегда привязано к узлу DOM. Чтобы добавить поведение, находят элемент и вешают обработчик через `addEventListener()`:

```
<button id="myButton"></button>
```

```
const button = document.getElementById('myButton')

// Добавляем обработчик, который вызовется при клике на элементе button
button.addEventListener('click', () => alert('Boom 1!'))

// Добавляем второй обработчик
button.addEventListener('click', () => alert('Boom 2!'))
```

Обработчик — функция, которую браузер вызывает при наступлении события. Несколько обработчиков на одном элементе выполняются в порядке регистрации.

При необходимости обработчик снимают — на практике это случается редко:

```
const button = document.getElementById('myButton')

const handler = () => alert('Boom 1!')
button.addEventListener('click', handler)

// Важно, что сюда передается ровно та же самая функция по ссылке
button.removeEventListener('click', handler)
```

События можно инициировать из кода. Например, [focus](https://developer.mozilla.org/en-US/docs/Web/API/HTMLElement/focus) переводит фокус на поле ввода — удобно в чатах, где курсор должен сразу оказаться в строке сообщения:

```
const input = document.getElementById('textInput')
// Курсор появляется в поле для ввода
// Попробуйте поменять канал
// В большинстве мессенджеров вы увидите, что поле для ввода фокусируется автоматически
input.focus()
```

## Как тестировать

События лучше всего отрабатывать прямо в браузере:

1. Откройте консоль на любом сайте и найдите элементы, с которыми хотите экспериментировать — кнопки, формы.

1. Для начала подойдёт `document.body`.

1. Повесьте обработчик.

1. Вызовите действие и посмотрите на вывод.

```
const element = document.body
element.addEventListener('click', () => console.log('wow!'))
// Теперь можно кликать по любой части сайта
wow!

// Как быть, если нужен элемент формы?

// Выбираем первый встреченный input
const element2 = document.querySelector('input')
// Вешаем обработчик и выводим на экран все, что нужно
element2.addEventListener('keyup', () => console.log('pressed!'))
```

Так можно проверять разные типы событий на живых страницах.

## Объект события

У каждого события есть объект с полями, зависящими от типа.

У `click`, например, есть координаты щелчка. Объект передаётся обработчику первым аргументом:

```
<div id="myElement">Бум!</div>
```

```
const button = document.getElementById('myElement')

button.addEventListener('click', (event) => {
  // Обычный объект
  console.log(event)
  // Координаты точки, в которой произошел клик
  console.log(event.clientX)
  console.log(event.clientY)
})
```

На каждый клик создаётся новый `event` со своими значениями.

Полный список свойств удобно смотреть в консоли и в [документации](https://developer.mozilla.org/ru/docs/Web/API/Event).

Общие поля:

- `event.target` — элемент, на котором событие сработало; через него часто читают данные из разметки

- `event.type` — имя события (`click`, `keyup` и т. д.)

Пример: валидация пароля при вводе — подсветить поле красной рамкой, если длина меньше порога. Реализуйте обработчик `input` или `keyup`, читайте `event.target.value` и меняйте класс или стиль поля.

## Действие по умолчанию

У многих элементов браузер выполняет **действие по умолчанию** без участия вашего кода. Клик по ссылке ведёт на URL из `href`, даже если на элементе есть свой обработчик.

Отменить стандартное поведение можно через `event.preventDefault()`:

```
<a href="#" id="myElement">Бум!</a>
```

```
const button = document.getElementById('myElement')

button.addEventListener('click', (e) => {
  // Если этого не сделать, то браузер выполнит загрузку новой страницы
  e.preventDefault()
  alert(e.target.textContent)
})
```

Типичные случаи:

- клик по ссылке — переход по `href`

- кнопка `type="submit"` — отправка формы

- прокрутка колёсиком в `textarea` при переполнении

- правый клик — контекстное меню

## Конкуренция между событиями

Пока выполняется один обработчик, могут возникать новые события — от пользователя или из кода. Некоторые действия порождают цепочку (например `mouseup` и затем `click`), но обработка идёт через очередь, а не «вклиниванием».

Исключение — часть программно вызванных действий вроде `focus`, которые могут обработаться сразу.

Пока синхронный обработчик крутится (тяжёлые вычисления), интерфейс не отвечает. Долгие синхронные задачи в обработчиках — плохая идея. Асинхронная работа (сетевой запрос) не блокирует очередь событий так же жёстко.

После загрузки страницы обычно регистрируют обработчики и дальше ждут ввода — вся логика крутится вокруг событийной модели.

## Частые ошибки

Новички иногда передают в `addEventListener` не функцию, а результат её вызова:

```
const button = document.getElementById('myElement')
const handler = () => {
  alert('Click!')
}

button.addEventListener('click', handler())
```

`handler()` выполнится сразу при подписке. В обработчик попадёт возвращаемое значение функции, а не сама функция.
