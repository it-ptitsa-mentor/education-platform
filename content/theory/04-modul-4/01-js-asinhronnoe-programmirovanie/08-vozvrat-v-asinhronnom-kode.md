# Возврат в асинхронном коде

Подробнее о возврате из асинхронных функций. Мы уже видели: они не возвращают результат операции напрямую:

```
import fs from 'fs'

const noop = () => {}
const content = fs.readFile('./myfile', 'utf-8', noop)
console.log(content) // undefined
```

Результат доступен только в колбеке. Что если сделать `return` внутри колбека?

```
import fs from 'fs'

const content = fs.readFile('./myfile', 'utf-8', (_error, data) => {
  // что-нибудь делаем
  return data
})
console.log(content) // undefined
```

Ничего не меняется — этот `return` никто снаружи не читает. `return` в асинхронном коде полезен как **прерывание** выполнения, а не как способ вернуть данные вызывающему коду:

```
import fs from 'fs'

const content = fs.readFile('./myfile', 'utf-8', (_error, data) => {
  if (data === '') {
    return
  }
  // делаем что-нибудь с данными
})
console.log(content) // undefined
```

Это guard expression.

То же для своих асинхронных функций: асинхронной считается любая функция с хотя бы одной асинхронной операцией — даже если рядом есть синхронная обработка строк. Такая функция принимает колбек: иначе нельзя ни упорядочить шаги, ни сообщить о завершении.

Обёртка над чтением файла с `trim()`:

```
import fs from 'fs'

const readFileWithTrim = (filepath, cb) => {
  fs.readFile(filepath, 'utf-8', (_error, data) => {
    // При вызове колбек-функции передаем null первым параметром
    // Поскольку обработку ошибок мы пока не рассматриваем
    cb(null, data.trim())
  })
}

readFileWithTrim('./myfile', (_error, data) => console.log(data))
```

Цепочка рекурсивна: функция, вызывающая асинхронную, сама становится асинхронной и тоже принимает колбек. Иначе нельзя ни использовать результат (он приходит в другом стеке), ни узнать, успешно ли завершилась операция. Подробнее — в следующих уроках.
