# Асинхронный код

*Примеры опираются на файловую систему — так нагляднее всего видна асинхронность. Те же принципы работают на фронтенде и бэкенде.*

В синхронном коде функция выполняется там, где вызвана, и сразу при вызове. В асинхронном вызов не означает немедленного выполнения и не задаёт момент завершения. Копирование файла синхронно:

```
import fs from 'fs'

// Обязательно передавать вторым параметром `utf-8`,
// только тогда данные возвратятся в строковом представлении
const content = fs.readFileSync('./myfile', 'utf-8')
fs.writeFileSync('./myfile-copy', content)
```

В Node.js модуль `fs` с суффиксом `Sync` блокирует поток: ОС читает файл, затем пишет копию; между строками выполнение стоит.

Файловые операции на порядки медленнее обычного вызова функции. Пока ядро отвечает, процесс простаивает — синхронный подход неэффективен. Асинхронный код продолжает работу во время IO и узнаёт о завершении отдельно. Хорошо написанная асинхронная программа часто быстрее; иногда синхронный вариант задачу не потянет.

*IO (Input/Output) — ввод-вывод: файлы, сеть (тоже файловые дескрипторы), вывод на экран.*

Попробуем асинхронный `readFile`:

```
import fs from 'fs'

// пустая функция, чуть позже разберём её смысл,
// но асинхронная версия readFile требует передачи функции третьим параметром
const noop = () => {}
const content = fs.readFile('./myfile', 'utf-8', noop)
console.log(content)
```

Такой API невозможен: операция не завершается в месте вызова, значит нельзя вернуть её результат синхронно:

```
node index.js
undefined
```

Запомните: асинхронная функция не возвращает результат операции — новички часто путают её с синхронной.

Результат передают через **колбек** (callback) — функцию обратного вызова с сигнатурой `callback(error, result)`. Сначала ошибка (если была), затем данные. Про ошибки — в отдельном уроке.

```
import fs from 'fs'

// Вызовется тогда, когда выполнится асинхронная операция
const callback = (_error, data) => console.log(data)
// префиксом нижнего подчёркивания обозначается неиспользуемый аргумент

// readFile запускает на выполнение задачу чтения файла.
// Вторым параметром обязательно передать utf-8.
// Только в этом случае данные прочитаются в строковом виде.
fs.readFile('./myfile', 'utf-8', callback)
```

```
node index.js
content of file
```

После чтения рантайм вызывает колбек с содержимым. Проверим асинхронность:

```
import fs from 'fs'

const callback = (_error, data) => console.log(data)
console.log('before read')
// вызов функции не дожидается конца чтения файла,
// код сразу продолжит выполняться дальше
fs.readFile('./myfile', 'utf-8', callback)
console.log('after read?')
```

```
node index.js

before read
after read?
content of file
```

Колбек запускается, когда в текущем стеке вызовов не остаётся функций — то есть после завершения всего синхронного файла. У колбека свой стек.

Каждый колбек асинхронной функции может породить новый стек и новые асинхронные вызовы.

Node.js ждёт завершения всех запущенных асинхронных операций:

```
import fs from 'fs'

fs.readFile('./myfile', 'utf-8', (_error, data) => console.log('First!'))
fs.readFile('./myfile', 'utf-8', (_error, data) => console.log('Second!'))
```

Два чтения стартуют почти сразу. В каком порядке напечатаются колбеки?

```
node index.js

Second!
First!

node index.js

First!
Second!
```

Однозначного ответа нет. Параллельный запуск не упорядочивает завершение — для последовательности нужны вложенные колбеки (дальше по курсу).

Асинхронность сложнее синхронной: линейный текст, нелинейное исполнение. С ростом числа колбеков сложность растёт быстро; позже разберём промисы и *async/await*.
