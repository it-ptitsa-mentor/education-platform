# Объектная композиция

Здесь — как организовать код на классах без лишнего наследования.

*В языках без классической инкапсуляции такие проблемы реже. Попробуйте Clojure или Elixir — картина другая.*

Допустим, на сайте есть аутентификация. После входа показываем приветствие: до 18 лет — один текст, иначе — другой.

Здесь `if` был бы нормальным решением. Урок про полиморфизм в классовой модели — идём через подтипы. Задача намеренно простая.

Частый порыв — классы `Under18` и `Above18` с `getMessage()`:

```
// Классы Under18 и Above18 наследуют свойства и методы User
class Under18 extends User {
  getMessage() {
    // Hi Sam
    return `Hi ${this.firstName}`
  }
}

class Above18 extends User {
  getMessage() {
    // Hello Mr Smith
    // Hello Mrs Tomson
    return `Hello ${this.appeal} ${this.lastName}`
  }
}
```

```
// Где-то в шаблоне
// Правильный класс для пользователя выбирается на момент начала обработки http-запроса -.
= user.getMessage()
```

Работает, но путь тупиковый. Сегодня два возраста, завтра — отдельная логика после 65, потом деление по полу. Комбинаций станет много, под каждую — свой класс пользователя:

- девушки старше 18

- девушки младше 18

- парни старше 18

- парни младше 18

- ...

В учебниках любят транспорт: плавает, летает, ездит — а потом появляется амфибия.

Почему подтипы пользователя — плохая идея? Пользователь из предметной области. Приветствие на экране — логика приложения, не доменная модель. Если смешивать, в `User` постепенно окажется всё, что связано с сайтом, — [божественный объект](https://ru.wikipedia.org/wiki/%D0%91%D0%BE%D0%B6%D0%B5%D1%81%D1%82%D0%B2%D0%B5%D0%BD%D0%BD%D1%8B%D0%B9_%D0%BE%D0%B1%D1%8A%D0%B5%D0%BA%D1%82).

Правильнее композиция: объекты сотрудничают, иерархия не раздувается. Два варианта приветствия — два класса: *GreetingForAbove18* и *GreetingForUnder18*, у каждого `getMessage`:

```
class GreetingForUnder18 {
  getMessage(user) {
    return `Hi ${user.firstName}`
  }
}

class GreetingForAbove18 {
  getMessage(user) {
    return `Hello ${user.appeal} ${user.lastName}`
  }
}
```

Передать пользователя в конструктор или в метод? Пользователь — абстракция данных с идентичностью и жизненным циклом. Вывод строки — операция без состояния; класс здесь ради полиморфизма. Лучше аргумент метода:

```
// Где-то в шаблоне
= greeting.getMessage(user)
```

Выбор объекта — фабрика до рендера шаблона:

```
const buildGreetingObject = (user) => {
  if (user.getAge() < 18) {
    return new GreetingForUnder18()
  }
  else {
    return new GreetingForAbove18()
  }
}
```

Пользователь остаётся пользователем — только доменная логика. Десять вариантов приветствия — десять классов приветствия, не десять подтипов `User`.

Новые задачи (письма после регистрации и т.д.) снова выносятся в отдельные классы + фабрика + полиморфный вызов — пользователь не трогается.

*Знакомо? Это та же Стратегия.*

Появляется много маленьких классов — по одному на вариант поведения. Часто без собственного состояния, только ради полиморфного вызова.
