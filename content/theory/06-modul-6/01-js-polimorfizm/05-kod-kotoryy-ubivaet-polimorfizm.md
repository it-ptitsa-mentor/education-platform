# Код, который убивает полиморфизм

## Формирование объектов

Посмотрите на функцию ниже: можно ли назвать её полиморфной?

```
const sayHiByEmail = (user) => {
  const sender = new EmailSender()
  // Отправляем email пользователю
  sender.send(user.getEmail(), 'hi!')
}
```

С одной стороны да: пользователь приходит снаружи, его можно подменить другим классом. С другой — внутри жёстко создаётся `EmailSender`, и без правки функции его не заменить.

Вывод простой: полиморфизм подтипов работает, когда объект передаётся в функцию извне, а не собирается внутри неё.

*Объект внутри функции допустим, если имя класса задаётся динамически — это разберём в уроке про метапрограммирование.*

## Проверка типов

Ещё пример. Есть ли здесь полиморфизм?

```
const sayHi = (user) => {
  if (user instanceof User) {
    console.log(`Hello, ${user}!`)
  }
  else if (user instanceof Guest) {
    console.log('Hello, guest!')
  }
  else {
    console.log('Who are you?')
  }
}
```

Объект снова снаружи, но внутри проверяется конкретный тип — поведение задаёт функция, а не объект. Она привязана к перечисленным классам и при их смене потребует правок. Полиморфизма подтипов нет.

`instanceof` иногда уместен, чтобы не усложнять код, но чаще сигнализирует о слабом дизайне — такой стиль плохо стыкуется с современным ООП.

Варианты для задачи выше:

- Перенести логику в классы: `user.sayHi()`. Осторожно: легко получить [божественный объект](https://ru.wikipedia.org/wiki/%D0%91%D0%BE%D0%B6%D0%B5%D1%81%D1%82%D0%B2%D0%B5%D0%BD%D0%BD%D1%8B%D0%B9_%D0%BE%D0%B1%D1%8A%D0%B5%D0%BA%D1%82). Чаще лучше второй путь.

- Ввести методы вроде `isUser` и `isGuest`:
```
const sayHi = (user) => {
  if (user.isUser()) {
    console.log(`Hello, ${user}!`)
  }
  else if (user.isGuest()) {
    console.log('Hello, guest!')
  }
  else {
    console.log('Who are you?')
  }
}
```

  Объёма кода не меньше, зато зависимость от методов, а не от типов. Смена структуры классов не затронет функцию, если контракт методов сохранится.
