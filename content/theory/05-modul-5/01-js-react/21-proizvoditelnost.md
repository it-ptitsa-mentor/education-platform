# Производительность

> Преждевременная оптимизация — корень всех зол.

Перед разговором о скорости полезно прочитать [Optimization.guide](http://optimization.guide/).

Стоит помнить: виртуальный DOM — уже встроенная оптимизация, и React «из коробки» достаточно быстр, чтобы долго не думать о производительности. Многим проектам этого хватает на весь срок жизни.

В общих чертах React работает так:

1. Монтирование (mount) вызывает рендеринг приложения.

1. Получившийся DOM вставляется в реальный DOM целиком, так как там еще ничего нет. А виртуальный DOM, в свою очередь, сохраняется внутри React для последующего обновления.

1. Изменение состояния приводит к вычислению нового виртуального DOM.

1. Вычисляется разница между старым виртуальным DOM и новым.

1. Разница применяется к реальному DOM.

## Согласование

При каждом изменении состояния компонента запускается reconciliation — сравнение старого и нового дерева. В общем случае полный поиск отличий в двух деревьях имеет сложность **O(n³)**.

Если события приходят часто, а виртуальное дерево выросло, лаги становятся заметны без профилировщика.

React просит задавать стабильный атрибут `key` для элементов списков. Это снижает сложность diff до **O(n)**.

Требование ключей проверяет сам React: в консоли появятся предупреждения, если их нет.

## Рендеринг

На практике полный пересчёт виртуального DOM на любое изменение дорог. Поле ввода означает новый виртуальный DOM на каждый символ. В крупном форуме с глубоким деревом полный рендер при наборе текста даёт заметную задержку — DOM при этом часто почти не меняется.

В React Developer Tools есть режим подсветки компонентов при перерисовке. Без специальных мер на событие часто перерисовывается всё дерево, хотя меняется малая часть.

React умеет пропускать компоненты без изменений, если они ведут себя как чистые функции.

При обновлении вызывается цепочка:

1. `getDerivedStateFromProps()`

1. `shouldComponentUpdate()`

1. `render()`

1. `getSnapshotBeforeUpdate()`

1. `componentDidUpdate()`

Перерисовку можно остановить в `shouldComponentUpdate()`: при `false` компонент не рендерится. Достаточно проверить, что не изменились `props` и `state`:

```javascript
shouldComponentUpdate(nextProps, nextState) {
  return !shallowEqual(this.props, nextProps)
    || !shallowEqual(this.state, nextState);
}
```

`shallowEqual()` сравнивает только верхний уровень объектов. Отсюда видно, почему нельзя мутировать состояние напрямую: `this.state.mydata.key = 'value'` — ссылка та же, содержимое другое, сравнение не сработает.

Большинство компонентов ведут себя как чистые функции; состояние часто в корне. Вместо `React.Component` можно наследовать `React.PureComponent` с готовым `shouldComponentUpdate`.

Пример можно открыть в CodePen с подключённым React. При клике корень перерисовывается, вложенный — нет.

Но `PureComponent` легко сломать незаметно.

### Пропсы по умолчанию

Ловушка — значения по умолчанию в JSX:

```javascript
class Table extends React.PureComponent {
  render() {
    const { options } = this.props
    return (
      <div>
        {this.props.items.map(i =>
          <Cell data={i} options={options || []} />,
        )}
      </div>
    )
  }
}
```

`options || []` при каждом рендере создаёт новый массив (`[] === []` — false). `<Cell>` перерисуется без реальных изменений данных.

Вывод: используйте встроенный механизм для свойств по умолчанию.

### Колбеки

```javascript
class App extends React.PureComponent {
  render() {
    return (
      <MyInput
        onChange={e => this.props.update(e.target.value)}
      />
    )
  }
}
```

На каждый `render` создаётся новая функция-обработчик — снова ломается мемоизация. Выход — объявить обработчик на уровне класса:

```javascript
class App extends React.PureComponent {
  handleChange = (e) => {
    this.props.update(e.target.value)
  }

  render() {
    return (
      <MyInput
        onChange={this.handleChange}
      />
    )
  }
}
```
