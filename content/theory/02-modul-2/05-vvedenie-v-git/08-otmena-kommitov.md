# Отмена коммитов

- [Git revert](https://buildin.ai/4e68faee-48a6-4c40-86b0-a252e17414e4#cb1d07fa-4d75-48cb-be5c-c8125f20deb0)

- [Выводы](https://buildin.ai/4e68faee-48a6-4c40-86b0-a252e17414e4#483b7238-bad4-4e01-8df8-df8197dc41e8)

В Git не страшно ошибаться — наоборот, ошибки часто лучший способ научиться. Почти всегда можно восстановить или поправить коммиты; в крайнем случае поможет повторный `git clone`.

Что делать, если коммит уже сделан, но не устраивает? Так бывает регулярно, и у опытных разработчиков тоже:

- Забыли добавить в коммит нужные файлы

- Изменения нужно «откатить», чтобы доработать

- Изменения больше не актуальны, и их нужно удалить

- Изменения были сделаны по ошибке, и их нужно отменить

Git в основном движется «только вперёд». Здоровый подход — создавать новое, а не переписывать старое.

Все перечисленное можно решить новым коммитом с нужными правками. Это удобно и безопасно. Переписывание истории опасно при синхронизации с удалённым репозиторием — об этом позже.

Тем не менее в Git есть команды, упрощающие отмену и правку коммитов. Они делают историю понятнее, а откат — быстрее. Разберём их в этом уроке.

## Git revert

Простейший случай — отмена изменений. По сути это ещё один коммит с правками, противоположными отменяемому:

Вручную такой коммит собрать непросто, поэтому есть `git revert`:

```
# Этой команде нужен идентификатор коммита
# Это коммит, которым мы удалили файл PEOPLE.md
git revert aa600a43cb164408e4ad87d216bc679d097f1a6c
# После этой команды откроется редактор, ожидающий ввода описания коммита
# Обычно сообщение revert не меняют, поэтому достаточно просто закрыть редактор
[main 65a8ef7] Revert "remove PEOPLE.md"
 1 file changed, 1 insertion(+)
 create mode 100644 PEOPLE.md
# В проект вернулся файл PEOPLE.md

git log -p

commit 65a8ef7fd56c7356dcee35c2d05b4400f4467ca8
Author: tirion <tirion@got.com>
Date:   Sat Sep 26 15:32:46 2020 -0400

    Revert "remove PEOPLE.md"

    This reverts commit aa600a43cb164408e4ad87d216bc679d097f1a6c.

diff --git a/PEOPLE.md b/PEOPLE.md
new file mode 100644
index 0000000..4b34ba8
--- /dev/null
+++ b/PEOPLE.md
@@ -0,0 +1 @@
+Haskell Curry
```

`revert` отменяет не только последний коммит, но и любой из истории. Без системы контроля версий такого не добиться.

## Команда `git reset`

Допустим, нужно убрать только что сделанный по ошибке коммит. Подойдёт и `git revert`, но история станет менее читаемой. Если коммит свежий и ещё не уходил на GitHub, лучше сделать вид, что его не было.

Git умеет удалять коммиты — это опасно и уместно только для новых коммитов, которых нет ни у кого, кроме вас.

Если коммит уже на GitHub, историю менять нельзя — сломается работа у коллег.

Для удаления коммита — `git reset`:

```
# Добавляем новый коммит, который мы сразу же удалим
echo 'test' >> INFO.md
git add INFO.md
git commit -m 'update INFO.md'

[main 17a77cb] update INFO.md
 1 file changed, 1 insertion(+)
 # Важно, что мы не делаем git push

git reset --hard HEAD~

HEAD is now at 65a8ef7 Revert "remove PEOPLE.md"

# Если посмотреть `git log`, то последнего коммита там больше нет
```

У `git reset` много флагов и режимов: удаление коммитов, отмена без удаления, восстановление файлов из истории и т.д. Это уже продвинутый уровень; здесь — только база.

Флаг `--hard` — полное удаление. Без него `git reset` отменит коммит, но не сотрёт изменения: они останутся в рабочей директории для доработки.

`HEAD~` — «на один коммит назад от последнего». Здесь тильда `~`, её легко спутать с дефисом `-`. Два последних коммита — `HEAD~2`:

`HEAD` («голова») — последний коммит. Терминологию разберём в уроке про устройство Git.

Без `--hard` `git reset` по умолчанию в режиме `--mixed`: изменения остаются в рабочей директории, но выходят из индекса (unstage). Их можно поправить или отменить и закоммитить заново:

```
echo 'no code no pain' >> README.md
git add README.md
git commit -m 'update README.md'

[main f85e3a6] update README.md
 1 file changed, 1 insertion(+)

# Теперь откатываем последний коммит
git reset HEAD~

Unstaged changes after reset:
M   README.md

git status

On branch main
Your branch is up to date with 'origin/main'.

Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
    modified:   README.md
```

Опция `--soft` сохраняет изменения в индексе:

```
git reset --soft HEAD~1

git status

On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
    modified:   README.md
```

`git reset --soft` отменяет коммит, но оставляет изменения в индексе (staging area). Следующий коммит включит те же правки, если их не менять.

Последнего коммита больше нет, а сами изменения на месте — в рабочей директории для доработки.

## Выводы

- В Git почти всегда можно исправить ошибку — несколькими способами

- `git revert` — безопасная отмена: новый коммит, отменяющий старый. Удобно в команде, история не переписывается

- `git reset` — мощный, но рискованный инструмент:
  - `git reset --soft` оставляет изменения в индексе
  - `git reset --mixed` (по умолчанию) снимает с индекса, но оставляет в рабочей директории
  - `git reset --hard` полностью удаляет изменения без простого восстановления

- Историю меняют осторожно. Если коммиты уже на удалённом репозитории, лучше `git revert`; `git reset` ломает синхронизацию

- В Git логичнее двигаться вперёд: ошибку проще исправить новым коммитом, чем переписывать прошлое

- Перед `git reset --hard` убедитесь, что ничего ценного не потеряете

Git даёт гибкие инструменты — но нужно понимать последствия.
