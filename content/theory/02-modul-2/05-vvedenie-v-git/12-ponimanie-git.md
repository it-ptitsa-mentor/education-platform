# Понимание Git

Обилие команд пугает новичков. Git — комбайн с массой возможностей. Нужно ли знать всё? Как запомнить? Как научиться пользоваться эффективно? Ответы — в этом уроке.

Почти никто не знает Git до конца. Разработчики постоянно находят новое, гуглят команды и читают документацию. Git учат использованием — особенно в команде. В одиночку многие сложные ситуации просто не встречаются.

Чтобы освоиться, важно понять устройство и идеи Git. Тогда станет ясно: большинство команд — обёртки над простой моделью.

Выведем коммиты *my-git* с опцией `--graph`:

```
git log --graph

# Неполный вывод, чтобы не отвлекаться от сути
* commit e7bb5e51f96e572084f6c04ba3312e32ce6b8c0f (HEAD -> main, origin/main, origin/HEAD)
|
|     update README.md
|
* commit 65a8ef7fd56c7356dcee35c2d05b4400f4467ca8
|
|     Revert "remove PEOPLE.md"
|
|     This reverts commit aa600a43cb164408e4ad87d216bc679d097f1a6c.
|
* commit 5120bea3e5528c29f8d1da43731cbe895892eb6d
|
|     add new content
|
* commit e6f625cf8433c8b1f1aaed58cd2b437ec8a60f27
|
|     add INFO.md
|
* commit 273f81cf2117044f1973ea80ce1067a94bea3f80
|
|     remove NEW.md
```

Полоска слева — связи между коммитами. Каждый новый опирается на предыдущий. В терминах информатики это **односвязный список**: элемент ссылается на предыдущий; последний — **голова** (*head*).

В Git элементы — коммиты. Новый коммит ссылается на предыдущий; тот — на свой, до первого, который ни на что не ссылается.

«Голова списка» (`HEAD`) в Git явная и используется в операциях. Удаление последнего коммита:

```
# HEAD~1 означает: взять голову и удалить один коммит, начиная от нее
# То есть мы удаляем только последний коммит
git reset --hard HEAD~1
```

Сам список коммитов тоже имеет имя — *main*. В Git это **ветка** (*branch*). Отсюда и `git branch` для текущего места:

```
git branch

* main
```

Строго говоря, ветка — подвижный указатель на коммит. При каждом новом коммите указатель сдвигается вперёд.

Суть работы Git — строить односвязный список коммитов. Большинство команд ходят по списку, читают и меняют его: мы блуждаем по истории, правим её и добавляем коммиты.

## Ветки

Дальше видно: цепочка коммитов — не просто список, а **направленный ациклический граф** — много списков, переплетённых вместе.

Два человека делают длинные задачи по несколько дней. *main* должна оставаться рабочей — промежуточные правки туда коммитить нельзя, код может сломаться. Но коммитить всё равно нужно: небезопасно копить изменения только в рабочей директории. Выход?
