## Как добавить репозиторий

Сохранённый репозиторий можно в любой момент восстановить и продолжить с последнего коммита — это спасает, если локальную копию удалили или испортили.

Шаги:

1. Зарегистрируйтесь на GitHub и настройте SSH-ключи по инструкции. SSH — удобный и безопасный способ работы с GitHub

1. Создайте на GitHub репозиторий *my-git*. Он должен быть пустым — не включайте галочки с автогенерацией файлов

1. На странице репозитория GitHub покажет команды для привязки локального репозитория:
<!-- IMG (из Buildin, перезалить отдельно) -->
  Выполните их:

```
# Подробнее эти команды мы разберем позже
# Вам нужна ссылка для работы с репозиторием по SSH
git remote add origin git@github.com:<ИМЯ НА ГИТХАБЕ>/my-git.git
git branch -M main
git push -u origin main
```

1. Обновите страницу на GitHub. Изучите интерфейс и файлы. Каталога `.git` там нет — почему так, разберём позже

После `git push` репозиторий на github.com «связывается» с локальным *my-git*. Вопрос «разве это не один репозиторий?» — нет, это разные копии.

Git — **распределённая** система контроля версий: нет единого «главного» сервера, с которым все работают только локально.

Репозиторий на GitHub и на машине разработчика — полноценные копии. Git синхронизирует их общей историей и обменом коммитами:

<!-- IMG (из Buildin, перезалить отдельно) -->
В примере изменения на удалённый репозиторий отправляет `git push`.

После привязки локальная и удалённая копии совпадают. В работе они снова разойдутся — нужно **синхронизировать**: пушить свои коммиты и забирать чужие.

На GitHub коммиты попадают только после `git push`. Бывает, что локальный репозиторий удалили, забыв **запушить** изменения.

Дальше скачаем репозиторий с GitHub, как будто локальной копии нет. Удалите каталог *my-git* с компьютера.

## Как клонировать репозиторий

Репозитории на GitHub бывают публичными и приватными. Публичный любой может **клонировать** себе и работать как с собственным — но **push** в чужой репозиторий без прав недоступен.

Клонирование — базовая операция: рабочие проекты обычно лежат на GitHub (или аналогах), и для работы их клонируют локально.

Команда — `git clone`. Ссылку берут на странице репозитория: кнопка Code → вкладка SSH:

```
git clone git@github.com:<ИМЯ НА ГИТХАБЕ>/my-git.git
cd my-git
ls -la

# Если эта операция проходит первый раз,
# То вы можете увидеть такое подобное сообщение
The authenticity of host github.com cannot be established. RSA key fingerprint is SHA256: хххххххххх Are you sure you want to continue connecting (yes/no/[fingerprint])? yes Warning: Permanently added github.com (RSA) to the list of known hosts.
# Наберите yes и нажмите Enter
```

Получили копию репозитория, как до удаления *my-git*.

## Как получить изменения с GitHub

Разработчики не только отправляют коммиты, но и **забирают** их — чаще всего правки коллег, но иногда и со второго своего компьютера.

Перед работой имеет смысл выполнить `git pull --rebase`: подтянуть новые коммиты с удалённого репозитория в локальный.

Достаточно и `git pull`, но так иногда появляются лишние merge-коммиты и история становится шумнее.

Корректная работа с `git pull` связана с ветками и `git rebase` — это сложнее для новичков; вернёмся к теме позже.

## Выводы

Итог урока: локальный репозиторий с коммитами выложен на GitHub, его можно клонировать.

Польза на этом этапе: резервная копия на GitHub — код не страшно потерять локально, им можно поделиться.

GitHub — самый известный хостинг, но не единственный: Bitbucket, GitLab; GitLab можно развернуть у себя в компании.

### Самостоятельная работа

1. Выполните все шаги из урока, создайте и добавьте ssh-ключи на GitHub по инструкции

1. Добавьте файл *NEW.md* с произвольным текстом и закоммитьте

1. Отправьте изменения: `git push`

1. Обновите страницу на GitHub — должен появиться последний коммит
