# Хеш-таблицы

**Ассоциативный** массив — абстрактный тип данных для пар «ключ — значение». Его ещё называют словарём или мапой (от map). В разных языках ему соответствуют разные встроенные типы. В JavaScript эту роль играет объект (Object); в других языках:

- Ruby — Hash

- Lua — Table

- Python — Dictionary

- Elixir/Java — Map

Зачем он нужен? Ассоциативные структуры повсюду в прикладном коде: ими удобно описывать составные сущности с множеством полей. По сути все предыдущие уроки про объекты в JavaScript были про использование Object как ассоциативного массива.

В отличие от обычного (индексированного) массива, ассоциативный нельзя положить в память «как есть»: нет последовательных индексов, по которым лежат значения. Для реализации часто берут **хеш-таблицу** — структуру, которая внутри опирается на индексированный массив и функцию хеширования ключей. Это не просто размещение данных, а алгоритм с правилами доступа.

*Дальше — как устроены ассоциативные массивы внутри. Это полезно, если хочется понимать цену удобства объектов в JavaScript, а не воспринимать их как «магию».*

Что происходит при выполнении:

```
const data = {}
data['key'] = 'value'
```

## Хеширование

Любая операция в хеш-таблице начинается с преобразования ключа в индекс обычного массива: сначала вычисляют хеш ключа, затем приводят его к индексу (часто через остаток от деления).

**Хеширование** — преобразование произвольных входных данных в строку (реже число) фиксированной длины. Функция, которая это делает, называется хеш-функцией; результат — хеш или хеш-сумма. Известные семейства: CRC32, MD5, SHA и их варианты.

```
// В JavaScript нет встроенной поддержки алгоритма хеширования crc32 (удобен для наглядности)
// Поэтому используется сторонняя библиотека
import crc32 from 'crc-32'

const data = 'Hello, world!' // Любые данные, которые мы хотим хешировать
const hash = crc32.str(data)

// Хеш всегда одинаковый для одних и тех же данных!
console.log(hash) // => -337197338
```

С хешированием разработчики сталкиваются часто. Например, идентификатор коммита в git *0481e0692e2501192d67d7da506c6e70ba41e913* — это хеш содержимого коммита.

Полученный хеш можно превратить в индекс, например остатком от деления:

```
// Это делается для того, чтобы индексы не были слишком большими
// Чем больше размер массива, тем больше памяти он занимает
const index = Math.abs(hash) % 1000 // по модулю
console.log(index) // => 338
```

### За кулисами

Рассмотрим добавление значения в ассоциативный массив (в JavaScript — тип Object). Программист пишет:

```
const data = {}
data['key'] = 'value'
```

За одной строкой — цепочка шагов. Ниже упрощённое описание на JavaScript для наглядности (в движке это уровень ниже):

```
// Для простоты показано на JavaScript, хотя в реальности всё это происходит на более низком уровне

// 1. Создание ассоциативного массива приводит к инициализации индексированного массива внутри интерпретатора.
const internal = []
// Во время присвоения значения `data['key'] = 'value'`, интерпретатор выполняет несколько действий:

// 2. Хеширует ключ. Результатом хеширования становится число.
const hash = crc32.str('key')
// 3. Число, полученное на предыдущем шаге, преобразуется в индекс массива.
const index = Math.abs(hash) % 1000
// В значение внутреннего индексированного массива, по найденному индексу, записывается еще один массив,
// первым элементом которого становится ключ `'key'`, а вторым значение `'value'`.
internal[index] = ['key', 'value']
```

Почему хранят пару `[ключ, значение]`, а не одно значение? Ответ связан с коллизиями — ниже.

Чтение:

```
const data = {}
data['key'] = 'value'
console.log(data['key']) // => "value"
```

```
// Для простоты показано на JavaScript, хотя в реальности всё это происходит на более низком уровне

// 1. Хешируется ключ. Результатом хеширования становится число.
const hash = crc32.str('key')
// 2. Число, полученное на предыдущем шаге преобразуется в индекс массива.
const index = Math.abs(hash % 1000)

// 3. Если индекс существует, то извлекается массив, который находился внутри, и возвращается наружу.
return internal[index] // ['key', 'value']
```

## Коллизии

Ключом может быть любая строка — множество возможных ключей по сути бесконечно. Выход хеш-функции имеет ограниченную длину, значит множество хешей конечно.

Отсюда следует: не для каждого ключа найдётся уникальный хеш. Рано или поздно разные ключи дадут один и тот же индекс — как если бы под одним слотом массива лежало несколько элементов. Это **коллизия**. Есть разные [способы разрешения коллизий](https://ru.wikipedia.org/wiki/%D0%A5%D0%B5%D1%88-%D1%82%D0%B0%D0%B1%D0%BB%D0%B8%D1%86%D0%B0#%D0%A0%D0%B0%D0%B7%D1%80%D0%B5%D1%88%D0%B5%D0%BD%D0%B8%D0%B5_%D0%BA%D0%BE%D0%BB%D0%BB%D0%B8%D0%B7%D0%B8%D0%B9) (открытая адресация, метод цепочек); реализации хеш-таблиц отличаются выбранной стратегией.

```
// Пример коллизии
// Хеш-функция возвращает одинаковый хеш для разных строчных данных!
crc32.str('aaaaa0.462031558722291') // 1938556049
crc32.str('aaaaa0.0585754039730588') // 1938556049
```

При **открытой адресации** ищут следующий свободный слот: для второго значения проверят `1938556050`, затем `1938556051` и так далее, пока не найдут пустой.

Коллизии встречаются чаще, чем кажется — см. [парадокс дней рождения](https://ru.wikipedia.org/wiki/%D0%9F%D0%B0%D1%80%D0%B0%D0%B4%D0%BE%D0%BA%D1%81_%D0%B4%D0%BD%D0%B5%D0%B9_%D1%80%D0%BE%D0%B6%D0%B4%D0%B5%D0%BD%D0%B8%D1%8F).
