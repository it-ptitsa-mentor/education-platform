# Lock-файл

Поговорим об обновлении зависимостей. Все сразу — `npm update`. Одну библиотеку — `npm update name`, где `name` — имя пакета. Как именно обновится версия, зависит от записи в `package.json`.

Варианты:

```
dependencies {
  "package1": "*",
  "package2": "1.3.5",
  "package3": "~2.3.4",
  "package4": "^2.3.4",
}
```

- `*` — любая версия; после обновления в `node_modules` окажется последняя `package1`

- `1.3.5` — жёсткая фиксация; обновить такую запись командами нельзя

Интереснее тильда (`~`). В [семантическом версионировании](https://semver.org/) patch (третья цифра) меняют при исправлениях без ломающих изменений — на практике это не всегда так, и код может зависеть от старых багов.

В проектах десятки и сотни зависимостей, обновления частые. Можно везде писать `*`, но мажорные апдейты ломают сборку. Можно зафиксировать всё вручную — тогда обновлять придётся самим.

Компромисс — тильда: автоматически тянутся только патчи. Была `~2.10.3`, в реестре появилась `2.10.5` — `npm update` её поставит. То же для `2.10.15`. Но `2.11.5` или `3.0.0` npm проигнорирует.

С `^` фиксируется мажор; минор и патч обновляются вместе.

### Lock-файл

При каждой установке создаётся или обновляется `package-lock.json`.

В `package.json` — прямые зависимости. У каждой — свои зависимости, и так далее (**транзитивные**). Отсюда **dependency hell** — «ад зависимостей».

Мы не фиксируем версии транзитивных пакетов вручную. Допустим, у нас зависимость `A` версии `1.3.2`, а у `A` в зависимостях `B` с `*`. `A` поставится как указано, а `B` — последняя из реестра.

Поведение недетерминировано: автор `B` выпустит несовместимую версию — сломается `A`, а с ним и проект.

Если полгода не трогать репозиторий, удалить `node_modules` и поставить заново — с высокой вероятностью что-то не взлетит: за время что-то обязательно сменит мажор.

Ручной учёт всех транзитивных версий в `package.json` теоретически возможен, но при пяти прямых зависимостях транзитивных — сотни, и они меняются.

Требовать жёсткие версии от всех авторов библиотек тоже не выход.

**Lock-файл** — автоматизированная фиксация всего дерева. Пример содержимого:

```
{
  "name": "yandex-co",
  "version": "0.1.4",
  "lockfileVersion": 1,
  "requires": true,
  "dependencies": {
    "JSONStream": {
      "version": "1.3.1",
      "resolved": "https://registry.npmjs.org/JSONStream/-/JSONStream-1.3.1.tgz",
      "integrity": "sha1-cH92HgHa6eFvG8+TcDt4xwlmV5o=",
      "requires": {
        "jsonparse": "1.3.1",
        "through": "2.3.8"
      }
    },
    "abab": {
      "version": "1.0.4",
      "resolved": "https://registry.npmjs.org/abab/-/abab-1.0.4.tgz",
      "integrity": "sha1-X6rZwsB/YN12dw9xzwJbYqY8/U4=",
      "dev": true
    },
    "acorn": {
      "version": "4.0.13",
      "resolved": "https://registry.npmjs.org/acorn/-/acorn-4.0.13.tgz",
      "integrity": "sha1-EFSVrlNh1pe9GVyCUZLhrX8lN4c="
    },
    ...
}
```

Первая установка формирует файл со всеми версиями, включая транзитивные. Дальнейший `npm install` ставит ровно то, что в lock-файле — даже если в реестре появились новые релизы. Повтор через месяц даст тот же результат.

Если заработало сегодня — с большой вероятностью заработает у коллег завтра.

Lock-файл не отменяет `update` для прямых зависимостей: при допустимом по диапазону обновлении подтянется новая версия, lock пересчитается — не забудьте закоммитить его.

Поведение lock-файла богаче, чем в этом обзоре. Подробности — в [официальной документации npm](https://docs.npmjs.com/cli/v10/configuring-npm/package-lock-json).

Как и любой сгенерированный артефакт, lock-файл не правят вручную — следующая генерация затрёт правки.

## Самостоятельная работа

- Склонируйте учебный проект *nodejs-package* (см. задания курса), выполните `npm update`. Изучите вывод `git diff`

- Поменяйте версию любого пакета в `package.json` на точную (без `^`) и выполните `npm install`. Доступные версии: `npm info <packagename>`, например `npm info eslint`
