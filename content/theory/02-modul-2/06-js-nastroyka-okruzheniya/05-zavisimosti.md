# Зависимости

В JavaScript, как и в других языках, есть встроенные функции и модули, упрощающие разработку. Вместе они образуют **стандартную библиотеку**. В Node.js это модули для криптографии, HTTP, файловой системы и многого другого. В браузере набор меньше, но он тоже есть.

```
// Пример встроенной функции
Math.round(5.34) // 5

// Встроенный в Node.js модуль для работы с файлами
import fs from 'fs'

// Читает содержимое файла
const data = fs.readFileSync('path/to/file')
```

Стандартной библиотеки недостаточно для всех задач. Можно писать недостающее самому, но тогда любой нетривиальный проект превратится в бесконечный марафон: типовых проблем слишком много.

В коммерческих проектах используют миллионы строк чужого кода со всего мира. Такой код распространяется в **пакетах** — в JavaScript-мире это принятое название библиотек.

Пакеты решают разные задачи: от мелких утилит для [дат](https://date-fns.org/) до фреймворков, задающих архитектуру. Очень популярна [lodash](https://lodash.com/docs/) — около сотни функций для массивов, строк и других типов. Примеры:

```
// По общепринятому соглашению, lodash импортируется под именем _
import _ from 'lodash'

// Пересечение массивов
_.intersection([2, 1], [2, 3]) // => [2]

// capitalize делает первую букву заглавной
_.capitalize('hello') // Hello
```

Отдельные функции кажутся мелкими, но в большом проекте их накапливается много — и хочется ещё и нормальную документацию.

Обратите внимание на импорт из внешних библиотек: имя пакета указывают без `./`. Так JavaScript понимает, что это пакет, а не файл *lodash.js* рядом с кодом.

```
// Так будет происходить поиск файла lodash.js в текущей директории
import _ from './lodash.js'

// Так импортируется код из пакета
import _ from 'lodash'
```

## Установка

Если импортировать *lodash* и сразу запустить код, программа упадёт: пакет не найден. Сторонний пакет нужно добавить в проект как **зависимость** — код проекта от него зависит. В *npm* для этого есть команда установки:

```
# Обязательно выполнять в корне проекта
# только тогда lodash будет лежать в правильном месте
npm install lodash
```

Команда делает три вещи. Скачивает пакет из реестра npm в *node_modules* в корне (директорию создаёт при необходимости). Добавляет запись в *package.json*. Создаёт или обновляет *package-lock.json*.

```
tree -L 1
.
├── node_modules
├── package.json
├── package-lock.json
└── index.js
```

Зависимости попадают под ключ `dependencies` — всё, что не входит в стандартную библиотеку.

```
"dependencies": {
  "lodash": "^4.17.15"
}
```

Ключ — имя пакета для импорта. Значение — версия на момент установки.

Когда Node.js видит:

```
import React from 'react'
```

он ищет модуль в *node_modules/react/...*

Про *package-lock.json* подробнее — в следующем уроке; сейчас — общая идея.

У зависимостей проекта есть свои зависимости, у тех — свои (**транзитивные**). Цепочка длинная; одни и те же пакеты могут встречаться в разных версиях. *package-lock.json* описывает **все** устанавливаемые пакеты с точными версиями — у всех разработчиков получается один и тот же набор. Файл создаётся при `npm install`; при его наличии для установки лучше `npm ci`:

```
# Если мы хотим в точности те же версии всех пакетов,
# какие были у остальных разработчиков этого проекта
npm ci
```

## Связь с GIT

Чаще работают не «с нуля», а с готовым репозиторием. Что из зависимостей хранить в git?

Код пакетов в репозиторий не кладут — *node_modules* в *.gitignore*. А *package.json* и *package-lock.json* — часть исходников: по ним Node.js узнаёт, что ставить.

Склонировали проект и сразу запустили — упадёт: *node_modules* нет. Нужно установить зависимости по манифестам:

```
# Выполняется в корне проекта
# Устанавливает все указанные зависимости
npm ci
```

Команда не меняет проект — только скачивает пакеты в *node_modules* по *package.json* и версиям из *package-lock.json*.

## Предупреждения

При установке *npm* печатает много сообщений, например:

```
npm WARN deprecated minimatch@2.0.10: Please update to minimatch 3.0.2 or higher to avoid a RegExp DoS issue
```

Чаще всего это не ломает приложение — просто сигнал об устаревших версиях или неподдерживаемых пакетах.

## Самостоятельная работа

1. Установите в проект my*-js* пакет *lodash*

1. Добавьте в файл *index.js* импорт функций из пакета *lodash* как в примерах выше

1. Добавьте в файл *index.js* строчку `console.log(_.last(['one', 'two']));`

1. Запустите код, убедитесь что выводится `"two"`

1. Добавьте изменения на Github. Проследите, чтобы *node_modules* была исключена из отслеживания

1. Изучите структуру директории *node_modules*

1. Найдите исходники пакета *lodash* на Github и сравните с содержимым *node_modules*. Структура может отличаться — это нормально

1. Ознакомьтесь с [руководством GNU Make](https://www.gnu.org/software/make/manual/make.html#Introduction). Добавьте в *Makefile* команду `install`, проверьте, что она работает. С её помощью будут устанавливаться зависимости. Добавьте в *README.md* описание этой команды.
