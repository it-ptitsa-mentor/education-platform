# Пайплайн

У процесса есть вход, у другого — выход; потоки можно подменять. Логичный шаг — **соединить** их. Такой приём называется **пайплайн** (*pipeline*), и в этом уроке он в центре внимания.

Пайплайн связывает программы и протаскивает через них данные: цепочка, где каждое звено фильтрует или преобразует поток.

При грепе мы искали одно слово. Часто нужно найти строки, где встречаются **несколько** слов — в любом порядке, но в одной строке.

Можно усложнять сам `grep`, но пайплайн даёт тот же эффект без тяжёлой программы:

```
grep PATH .profile | grep local

PATH="$HOME/.local/bin:$PATH"
```

Символ `|` — **пайп**: оболочка подключает STDOUT одной команды к STDIN следующей.

Утилиты, читающие файлы, обычно умеют читать и STDIN. `grep` на входе и выходе работает с текстом — значит, цепочку можно наращивать:

Запись `grep PATH .profile | grep local` можно переписать с перенаправлением — иногда так проще менять цепочку:

```
cat .profile | grep PATH | grep local
```

Здесь `cat` читает файл и отдаёт текст первому `grep`.

Ещё пример:

[Terminal session recording](https://asciinema.org/a/9Xb9BQi5GEWabm0bATlG2SOV7/iframe?cols=120&rows=17&preload=1)

```
cat source | grep Dog | uniq | sort
```

По шагам:

1. Читается файл *source*

1. Грепаются входные данные по подстроке `Dog`

1. Убираются дубли (в исходном файле две одинаковых строки `Dog`)

1. Сортируются входные данные и выводятся на экран

Пайплайн — основа unix-философии:

- Программа делает одну вещь и делает её хорошо

- Программы должны стыковаться друг с другом

- Универсальный интерфейс — текстовые потоки

Поэтому большинство утилит гоняют сырой текст: приняли на вход, отдали в STDOUT.

Сложное поведение собирают из простых кирпичиков — идея **стандартных интерфейсов**, как у конструктора Lego.
