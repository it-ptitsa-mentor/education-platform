# Теория: Инкапсуляция

Среди возможностей ООП есть одна базовая, которую большинство ассоциирует с объектным стилем целиком — инкапсуляция. Это объединение функций и данных в одной структуре, причём внутреннее состояние (данные) скрыто от внешнего мира (этот аспект разберём позже). Такие функции называют методами. Мы уже много раз с ними сталкивались — в JavaScript методы повсюду.

## Сложно

Перед подробным разбором инкапсуляции — коротко о терминологии и путанице в сообществе. Это важно, если вы уже читали разную литературу. Если следующий абзац пока непонятен — пропустите и вернитесь в конце курса.

В части источников инкапсуляцией называют только сокрытие данных (data hiding) от прямого доступа снаружи (через `private`, `protected` и т.п.). На собеседовании часто ждут именно это определение — но оно верно лишь частично. Несмотря на распространённость, полезно разделять объединение данных с методами и сокрытие этих данных. В Python, например, есть объединение без сокрытия; если добавить сокрытие — архитектура почти не изменится, а если разъединить данные и методы — переписывать придётся почти всё. Похожая картина в языках со сокрытием: убрать его — мало что поменяется, кроме необходимости аккуратнее работать с объектами.

Итог: инкапсуляция — и объединение, и сокрытие там, где оно есть; где сокрытия нет — остаётся объединение. В этом курсе под инкапсуляцией понимаем только объединение данных и функций, а сокрытие данных выделяем отдельно — чтобы не путать, что имеется в виду под термином.

Зачем нужно сокрытие данных — в уроке про инварианты.

## Конец Сложно :)

```
// Вызов метода
user.getName()

// Вызов функции
getName(user)
```

Как методы устроены внутри — в следующем уроке. Сейчас — внешние особенности.

### Автодополнение

Методы вместо свободных функций дают неожиданный эффект: в редакторах можно автодополнять вызовы методов у объекта. Это снижает нагрузку на память и радует разработчиков. Есть версия, что именно эта удобность сильно подтолкнула популярность ООП (не доказано, но правдоподобно).

*В языках с модулями автодополнение есть и у обычных функций — но сначала нужно верно указать модуль. Пример из Elixir: `User.getName(user)`. Есть языки с Unified Function Call (например Nim), где функции можно вызывать как методы и получать автодополнение.*

### Естественность

Другая черта спорная. Многим код с методами кажется «естественнее»: абстракции на данных, по их мнению, строятся только через методы; без объединения данных и функций абстракция невозможна. Такое ощущение часто от ограниченного опыта — разработчик не выходил за рамки популярных ООП-языков, где функциональные абстракции кажутся неестественными или невозможными.

Это не так. Достаточно пройти курс «JS: Абстракции с помощью данных», чтобы убедиться. Абстракции и моделирование существовали до ООП и будут существовать после.

*Представьте «добавление в друзья» в ООП-стиле: кто кого добавляет (первый второго или наоборот) и как не уйти в рекурсию при взаимном добавлении?*

### Методы не нужно импортировать

Третья особенность практичнее. С объектами не нужно отдельно импортировать методы, в отличие от свободных функций. Любая функция, получившая объект, может вызывать его методы. При работе только с функциями пришлось бы импортировать каждую нужную операцию. Плата за это — ограничения на расширение объектов (об этом в следующих уроках).

```
// send просто функция, поэтому ее надо импортировать перед использованием
import send from 'mail'

const sendEmail = (user) => {
  send('Subject', user.getEmail()) // Не надо импортировать getEmail потому что это метод
}
```

Если объекта нет (как в примере выше), языки и библиотеки решают по-разному. В JavaScript обычные функции и методы сосуществуют. То же в Python. В Ruby и PHP (в современных фреймворках) свободные функции менее «естественны», хотя их можно писать. В Java нельзя создать обычную функцию вне метода — любая функция привязана к объекту. Поэтому в Java объекты создают на каждый чих: код раздувается, простые вещи усложняются. В Elixir и Clojure методов в привычном смысле нет — и не нужны; код при этом лаконичный и расширяемый.

*В Java для имитации свободных функций используют статические методы — работа без создания экземпляра.*

### Цепочки вызовов

Четвёртая особенность — цепочки. Вспомните:

```
const brand = 'bmw'
// Этот метод не изменяет строку, а возвращает новую!
brand.toUpperCase() // BMW
```

Метод возвращает новую строку, у которой тоже есть методы — их можно вызывать дальше:

```
// Переменная, так как перезаписываем
let brand = 'bmw'
brand = brand.toUpperCase() // BMW
brand = brand.concat(' & Kia') // BMW & Kia
brand = brand.replace('BMW', 'Opel') // Opel & Kia
console.log(brand) // => Opel & Kia
```

А если не создавать промежуточные переменные?

```
const brand = 'bmw'
// Теперь лучше сделать новую константу, вместо переменной
const newBrand = brand.toUpperCase().concat(' & Kia').replace('BMW', 'Opel')
console.log(newBrand) // => Opel & Kia
```

Код короче и иногда понятнее. Но не увлекайтесь — легко уйти в нечитаемые однострочники. Разбить на несколько строк всегда можно:

```
const brand = 'bmw'
const newBrand = brand.toUpperCase()
  .concat(' & Kia')
  .replace('BMW', 'Opel')
console.log(newBrand) // => Opel & Kia
```

Цепочки работают и когда возвращается значение другого типа — применяются методы этого типа:

```
const brand = 'bmw'
const newBrand = brand.toUpperCase()
  .split('') // ['B', 'M', 'W']
  .reverse() // ['W', 'M', 'B']
  .join('') // WMB
console.log(newBrand) // => WMB
```

Такие цепочки называют [fluent interface](https://en.wikipedia.org/wiki/Fluent_interface).

Как и многое в современном ООП, цепочки не эксклюзивны. Они близки к пайплайну (pipeline). В командной строке вы наверняка видели:

```
# | называется пайпом (pipe - труба)
# Пайплайн часто сравнивают с бусинками (функции), через которые пропускают веревку (данные).
cat sample | grep -v a | sort -r
```

Команды передают данные слева направо через обработчики. Идея из математики, старше программирования. Во многих языках пайплайн — языковая конструкция (или макрос в Lisp). Её пытаются добавить и в JavaScript — [стадия рассмотрения](https://github.com/tc39/proposal-pipeline-operator). Пример:

```
// Так с функциями работать неудобно
const result = exclaim(capitalize(doubleSay('hello')));
console.log(result) // => "Hello, hello!"

// А вот это совсем другое дело (но нужно привыкнуть)
// |> – это пайп, он отправляет данные, пришедшие слева в функцию справа
const result = 'hello'
  |> doubleSay
  |> capitalize
  |> exclaim;

console.log(result) // => "Hello, hello!"
```
