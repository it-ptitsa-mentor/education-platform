# Теория: Исключения

Обработка ошибок — отдельная большая тема. До сих пор мы её обходили, но в реальных приложениях на тысячи и сотни тысяч строк она влияет на сопровождение, расширение и то, что увидит пользователь при сбоях.

В уроке — механизм исключений. Сначала — про ошибки вообще.

У строк есть `text.indexOf(str)`: ищет подстроку и возвращает индекс. Подстрока не найдена — это ошибка? Нет, штатный исход. В редакторе поиск без совпадений — обычная ситуация, программа не падает.

*Загляните в документацию: как функция сообщает, что подстроки нет?*

Другой случай — «открыть файл» в редакторе. Файл удалили во время открытия — это ошибка, но не баг программиста. Такое случается всегда; задача — корректно обработать, а не «запретить».

Насколько это критично? В плохих приложениях — падение целиком. В хороших — предупреждение, повтор или другое действие.

Одна ситуация на разных уровнях — и ошибка, и норма. Функция читает файл и не смогла — для неё это сбой. Должно ли ронять всё приложение? Нет — решает вызывающий код, не низкоуровневая функция.

## Коды возврата

В языках до ~1990 ошибки часто кодировали возвращаемым значением. В C при неудаче — `NULL` или отрицательное число с кодом:

```
int write_log()
{
    int ret = 0; // return value 0 if success
    FILE *f = fopen("logfile.txt", "w+");

    // Проверяем, получилось ли открыть файл
    if (!f)
        return -1;

    // Проверяем, что не достигли конца файла
    if (fputs("hello logfile!", f) != EOF) {
        // continue using the file resource
    } else {
        // Файл закончился
        ret = -2;
    }

    // Не получилось закрыть файл
    if (fclose(f) == EOF)
        ret = -3;

    return ret;
}
```

Заметьте проверки и `ret`. Каждая рискованная операция требует контроля и кода возврата.

Проблема: ошибку обрабатывают редко там, где она возникла. Функция *A* должна сообщить пользователю, а сбой в *E* по цепочке *A => B => C => D => E*. Все промежуточные функции обязаны знать коды и пробрасывать их — кода про ошибки становится больше, чем про задачу.

*Есть схемы на возвратах без этого минуса — например, монада Either.*

## Исключения

На этом фоне появились исключения: доставить ошибку из места возникновения в место обработки, минуя промежуточные уровни — стек раскручивается сам.

Две роли: где ошибка — **выбрасывают** (`throw`); где ловят — **перехватывают** (`catch`).

```
// Функция, которая может выбросить исключение
const readFile = (filepath) => {
  if (!isFileReadable(filepath)) {
    // throw – способ выбросить исключение
    throw new Error(`'${filepath}' is not readable`)
  }
  // ...
}

// Где-то в другом месте программы

const run = (filepath) => {
  try {
    // Функция, которая вызывает readFile. Возможно не напрямую, а через другие функции.
    // Для механизма исключений это не важно.
    readFile(filepath)
  }
  catch (e) {
    // Этот блок выполняется только в одном случае, если в блоке try было выброшено исключение
    // Любая обработка ошибки, например, вывод в консоль
    console.log(e)
  }
  // Если тут будет код, он продолжит выполняться
}
```

Исключения — объекты `Error` с сообщением, трассировкой стека и прочими полями.

Выброс — `throw`:

```
const e = new Error('Тут любой текст')
throw e // Исключение можно создать отдельно, а можно сразу же там, где используется throw
```

Можно бросать не только `Error`, но и любое значение:

```
const message = 'Тут любой текст'
throw message
```

`throw` обрывает выполнение сильнее `return`: не только текущую функцию, а весь путь до ближайшего `catch`.

`try catch` — два блока: перехват и обработка.

Первый блок после `try`:

```
try {
  // ...
}
```

Исключения из `try` попадают во второй блок; без ошибки `catch` пропускается.

```
catch (e) {
  // ...
}
```

Ошибка доступна в `e` (имя любое):

```
try {
  // ...
}
catch (myError) {
  console.log(myError)
}
```

В `catch` можно делать что угодно, в том числе бросать снова — поймает внешний `try`:

```
const myFunc = () => {
  try {
    // ...
  }
  catch (e) {
    throw new Error('new error')
  }
}

try {
  myFunc()
}
catch (e) {
  console.log(e) // => new error
}
```

`try/catch` часто на верхнем уровне, но бывают промежуточные перехваты с повторным выбросом — тема на опыт.
