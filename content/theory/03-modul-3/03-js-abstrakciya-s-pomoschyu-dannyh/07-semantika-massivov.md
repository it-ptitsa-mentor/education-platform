# Семантика массивов

В прошлом уроке точку представили массивом. Подходит ли это? Разберёмся.

У конструкций языка есть не только синтаксис, но и **семантика** — для каких задач их задумали. Инструменты часто применяют «не по назначению», и код становится труднее читать, отлаживать и сопровождать.

**Массив** — коллекция однотипных значений, которую обычно перебирают и обрабатывают одинаково. Элементы слабо связаны и могут существовать отдельно. Позиция элемента часто не фиксирована жёстко или зависит от задачи. Примеры, где массив уместен:

- Список стоп-слов

- Список пользователей

- Список уроков курса

- Список ходов шахматной партии (порядок важен)

В графической библиотеке массив подходит для набора точек или списка отрезков.

**Точка** — не коллекция. Это целое: координаты `x` и `y` не имеют смысла по отдельности, порядок между ними не как в списке пользователей. Код, ожидающий массив из двух элементов с фиксированными позициями, по сути использует массив как составную структуру из двух чисел.

Здесь уместнее **объект**:

```
const point = { x: 2, y: 3 }
const symmetricalPoint = { x: -point.x, y: point.y }
```

Строк кода чуть больше, но семантика яснее: вместо `point[0]` — `point.x`. При выводе сразу видно, о чём речь.

Отрезок на массивах — массив массивов:

```
const point1 = [2, 3]
const point2 = [-8, 10]
const segment = [point1, point2]

// код сложный для понимания
point1[1]
point2[0]
segment[1][0]
```

Без контекста непонятно, что это отрезок. Хорошие имена переменных помогают слабо.

Правильная под задачу структура важнее:

```
const point1 = { x: 3, y: 4 }
const point2 = { x: -8, y: 10 }
const segment = { beginPoint: point1, endPoint: point2 }

// такой код уже проще для понимания
point1.x
point2.y
segment.endPoint.y
```

Запомните: код, который заставляет гадать — неговорящие имена, слабые абстракции, неверные структуры, зависимость от контекста — плохой код (не путайте простоту с примитивностью).

Объект даёт **расширяемость**. Массив-структура хрупок: переставить аргументы нельзя — сломается код, рассчитанный на порядок. Расширить тоже рискованно: что-то продолжит работать, что-то нет (например, деструктуризация `[x, y] = point`). У объекта нет жёсткой позиции ключей; новый ключ чаще всего не ломает программу.

Что ещё моделировать объектом? Любую одиночную сущность:

- Пользователь

- Курс

- Урок

- Платёж

- Шахматная партия (помимо даты, имён и места, она содержит набор ходов)
