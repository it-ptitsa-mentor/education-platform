# Инварианты

Абстракция освобождает от деталей реализации и позволяет сосредоточиться на использовании. При необходимости внутренности можно переписать, не ломая вызывающий код. Есть и ещё причина применять абстракции — соблюдение **инвариантов**.

Инвариант в программировании — логическое условие, которое описывает согласованность состояния (набора данных).

Пример. После того как мы описали конструктор и селекторы рациональных чисел, неявно предполагались такие инварианты:

```
const num = makeRational(numer, denom)
numer === getNumer(num) // true
denom === getDenom(num) // true

// Или с учётом нормализации
// numer / denom === getNumer(num) / getDenom(num);
```

Передав в конструктор числитель и знаменатель, мы ожидаем получить их обратно через селекторы. Так проверяется корректность абстракции. По сути, этот фрагмент — почти готовые тесты.

Инварианты относятся к любой операции, и иногда они нетривиальны. Рациональные числа можно сравнивать, но не «в лоб»: одна дробь записывается по-разному — *1/2* и *2/4*. Код, который это игнорирует, ведёт себя неверно:

```
const num1 = makeRational(2, 4)
const num2 = makeRational(8, 16)

console.log(num1 === num2) // false
```

Приведение дроби к каноническому виду называют **нормализацией**: сокращение, знак, перенос знака в числитель. Реализовать можно по-разному. Проще всего нормализовать при создании, внутри `makeRational()`. Другой вариант — при каждом вызове `getNumer()` и `getDenom()`, но тогда вычисления повторяются. Это снимается [мемоизацией](https://ru.wikipedia.org/wiki/%D0%9C%D0%B5%D0%BC%D0%BE%D0%B8%D0%B7%D0%B0%D1%86%D0%B8%D1%8F).

С учётом нормализации инвариант конструктора и селекторов нужно уточнить: `getNumer()` и `getDenom()` возвращают не исходные аргументы, а значения после нормализации (если дробь уже нормализована, они совпадут с переданными).

```
const num = makeRational(10, 20)
getNumer(num) // 1
getDenom(num) // 2
```

Абстракция не только скрывает реализацию, но и гарантирует инварианты. Обход абстракции легко нарушает внутренние преобразования:

```
// Обход конструктора

// Эти данные не нормализованы, потому что не использовался конструктор
const num = { numer: 10, denom: 20 }

// Возвращается не то, что должно (ожидается нормализованный возврат):
getNumer(num) // 10
getDenom(num) // 20

// Прямая модификация

const num = makeRational(10, 20)
// тут не может быть нормализации, так как прямое изменение
num.numer = 40

getNumer(num) // 40
getDenom(num) // 2
```

Прямая работа с полями, минуя конструктор и селекторы, ломает инварианты, которые обеспечивала логика абстракции. Поэтому важно пользоваться API так, как задумано.

Возникает вопрос: можно ли сделать обход невозможным? Да — это **сокрытие данных** (data hiding). В языках для этого есть [специальный синтаксис](https://github.com/tc39/proposal-class-fields#private-fields). Защиту можно организовать и без него — через функции высшего порядка, замыкания и обмен сообщениями (подробнее в SICP).
