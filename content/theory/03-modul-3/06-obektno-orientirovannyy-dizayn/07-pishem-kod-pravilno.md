# Пишем код правильно

Имея за плечами базу по ООП, попробуем ответить: как **структурировать** код в классовых языках на практике.

В учебниках часто отсылают к **SOLID** как к рецепту «правильного» кода. На деле одних аббревиатур мало — без контекста они плохо подсказывают, что именно менять.

Возьмём **SRP** (Single Responsibility — «S» в SOLID):

> Должна быть ровно одна причина для изменения класса

Есть и другие формулировки, но суть та же. Проблема в общности: звучит как «делай хорошо — будет хорошо». Нет чёткого критерия «здесь уже нарушение». В статьях всё выглядит очевидно, потому что автор уже разрезал ответственности. В живом коде разные разработчики дают противоположные советы; остаётся личное чутьё.

Пример — библиотека дат и времени. С чего начать проектирование?

С **сценариев использования**: представьте, что API уже есть, и вы им пользуетесь (TDD как раз толкает к этому). Прежде чем писать классы, спросите: нужны ли здесь объекты и ООП?

«Текущее время» — операция с началом и концом. Объект для неё не обязателен; достаточно функции. Минимальный API:

```
import { getCurrentTime } from 'dateTimeLib'

const currentTime = getCurrentTime()
console.log(currentTime) // => 2023-04-14T20:13:40.432+05:00
```

Когда интерфейс ясен, можно реализовывать внутренности. Насколько важна внутренняя структура? Не критично: это локальная библиотека, её можно переписать позже, когда накопится опыт поддержки и использования. Главная мысль: **сначала граница и сценарии, реализация — потом**.

Так устроена, например, [luxon](https://github.com/moment/luxon/):

```
import { DateTime } from 'luxon'

console.log(DateTime.now().toISO()) // => 2023-04-14T20:13:40.432+05:00
console.log(DateTime.now().toFormat('MMMM dd, yyyy')) // => April 14, 2023
```

Объект можно создать, чтобы **запомнить конфигурацию** и не дублировать опции:

```
import { DateTime } from 'luxon'

const dateTime = new DateTime({ zoneName: 'Asia/Singapore', valid: true })
console.log(dateTime.toISO()) // => 2023-04-14T20:13:40.442+05:00
```

Сколько классов внутри — вопрос здравого смысла. Есть клиент (объект с конфигурацией, не с запросами/ответами) и результат `getCurrentTime`. Дальнейшее дробление часто не нужно; если понадобится — по **реальной** боли, а не из-за абстрактной «единственной ответственности». Сильный мотив для разделения — отделить **чистые** вычисления от кода с побочными эффектами.

Внутри библиотеки есть слой, который трогает систему (читает часы), и слой, который нормализует данные. Сначала отделяйте такие куски на уровне функций или методов. Любая операция, которую можно сделать чистой, — кандидат на вынос.

Тот же приём с **Reporter** из учебных примеров:

```
import Reporter from './Reporter.js'

// Делать ли Reporter абстракцией данных (то есть объектом, описывающим конкретный отчет)
// или нет, это большой вопрос.
// По умолчанию так делать не стоит, иначе весь код превратится
// в бесполезное (new Reporter('path/to/file'))->generate()
const reporter = new Reporter()
const report = reporter.generate('/path/to/report')
```

Заметьте: класс смешивает **грязную** работу (чтение с диска) и **чистую** (сбор отчёта). Не обязательно сразу переписывать всё, но это первое, на что смотреть: такой код сложнее тестировать. Если чтение вынести наружу, Reporter станет универсальнее — примет данные из файла, HTTP, формы:

```
import fs from 'fs'
import Reporter from './Reporter.js'

const reporter = new Reporter()
const data = fs.readFileSync('/path/to/report')
const report = reporter.generate(data)
```

Остальные буквы SOLID опираются на тему **полиморфизма в JavaScript** — разберём их там.
