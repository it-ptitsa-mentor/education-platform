# Объекты-Сущности, Объекты-Значения и внедренные объекты

## Объекты-Сущности (entities objects)

В учебниках ООП часто рисуют «сущности предметной области»: пользователи, заказы, товары. У такого подхода есть условия, без которых модель ломается.

*В учебниках это подаётся как главный аргумент за ООП, но в реальном коде большинство классов библиотек и фреймворков с предметной областью не связаны. Их держат ради полиморфизма и удобной структуры API.*

**Время жизни**. Такие объекты создают не на один вызов, а на период работы программы — часто дольше, в хранилище между запусками. Пользователь в системе — объект `User`: появляется при регистрации и живёт, пока его не удалят.

**Идентификация**. Чем один пользователь отличается от другого? Имя и фамилия не подходят: совпадения возможны, поля меняются. Для сущностей вводят **искусственный идентификатор** (обычно из БД) и сравнивают по нему.

```
class User {
  constructor(id, name) {
    this.id = id
    this.name = name
  }

  equals(user) {
    return this.id === user.id
  }
}

// С точки зрения нашей системы это один и тот же пользователь
// С точки зрения js – разные объекты
const user1 = new User(3, 'mike')
const user2 = new User(3, 'mike')

const user3 = new User(1, 'mike')

user1.equals(user2) // true
user1.equals(user3) // false
```

Похоже на сравнение объектов в JavaScript: равенство по ссылке, а не по полям. Два объекта с одинаковыми данными — всё равно разные экземпляры.

## Объекты-Значения (value objects)

Объекты с идентичностью и жизненным циклом — **объекты-сущности (entities)**. Есть и другой тип — **объекты-значения**.

Десять долларов в кошельке — не важно, какая именно купюра: меняете одну на другую того же номинала — сумма та же. То же с адресом доставки, страной, путём к файлу, URL, точкой на плоскости: важно **значение**, а не «какой именно объект его несёт».

В системе с разными валютами удобно описать деньги классом с номиналом и валютой. Сравнение строят по смыслу, а не по ссылке:

```
const m1 = new Money(150, 'usd')
const m2 = new Money(130, 'eur')

// Предположим, что 150 долларов по текущему курсу равны 130 евро
// Функция конвертирует деньги для сравнения
m1.equals(m2) // true
```

Объект здесь — оболочка для данных, а не идентификатор. Такие типы называют [объектами-значениями](https://martinfowler.com/bliki/ValueObject.html).

*На практике граница размыта: одна и та же вещь может быть и значением, и сущностью. Для большинства компаний деньги — значение; для типографии, печатающей купюры, важен уникальный номер каждой.*

Объекты-значения — осознанная абстракция. Для примитива часто хватает самого значения. Для составных вещей (точка, адрес, URL из частей) класс упрощает код. В JavaScript для URL есть встроенный класс:

```
import { URL } from 'url'

const url = new URL('/courses?page=2', 'https://ru.wiki.io')
url.host // 'ru.wiki.io'
url.pathname // '/courses'
url.searchParams.get('page') // '2'
```

Два одинаковых URL логично считать равными, но в стандартной библиотеке простой проверки равенства нет — типичный пример объекта-значения с недостающим API.

## Встраиваемые объекты (embedded objects)

Данные веб-приложений обычно лежат в реляционных БД: сущность — строка, поля — свойства. Иногда несколько полей описывают одну смысловую вещь, например адрес:

```
// Поиск из базы по идентификатору
// Гипотетический код
const user = User.find(5)

user.street // 'lenina'
user.zipcode // 432111
user.house // 10
```

**Первый подход** — вся логика адреса внутри сущности:

```
class User {
  // Где-то здесь конструктор и другие методы

  getFullAddress() {
    return `${this.street}, ${this.house}, ${this.zipcode}`
  }
}

user.getFullAddress()
```

Если адресные поля есть у нескольких сущностей, методы придётся дублировать.

**Второй подход** — отдельный класс и **внедрение** его в основной объект:

```
class Address {
  constructor(street, house, zipcode) {
    this.street = street
    this.house = house
    this.zipcode = zipcode
  }

  toString() {
    return `${this.street}, ${this.house}, ${this.zipcode}`
  }
}

class User {
  // Где-то здесь конструктор и другие методы

  getAddress() {
    // Так как у нас объект-значение,
    // то можно не боясь создавать его любое количество раз,
    // но в случае необходимости можно этот процесс оптимизировать
    return new Address(this.street, this.house, this.zipcode)
  }
}

user.getAddress().toString()
```
