# Подготовка данных

Тесты на одну функциональность часто похожи — особенно в части подготовки данных. В прошлом уроке каждый тест начинался с `makeStack()` — ещё не дублирование, но шаг к нему. В реальных проектах подготовка обычно тяжелее.

Допустим, мы разрабатываем библиотеку [Lodash](https://lodash.com/docs/4.17.21) и тестируем функции для коллекций:

- find

- filter

- includes

- и другие (всего их около 20 штук)

Для работы этих функций нужна заранее подготовленная коллекция. Проще всего придумать одну, которая подойдёт для тестирования большинства или даже всех функций:

```
import _ from 'lodash'
test('includes', () => {
  // Подготовили коллекцию coll
  const coll = ['One', true, 3, 10, 'cat', {}, '', 10, false]

  // Используем coll для тестирования
  expect(_.includes(coll, 3)).toBe(true)
  expect(_.includes(coll, 11)).toBe(false)
})
```

Если таких тестов десятки (в реальности — сотни), одна и та же коллекция начнёт кочевать по файлам.

Простой выход — вынести коллекцию на уровень модуля, вне тестовых функций:

```
import _ from 'lodash'

const coll = ['One', true, 3, 10, 'cat', {}, '', 10, false]

test('includes', () => {
  expect(_.includes(coll, 3)).toBe(true)
  expect(_.includes(coll, 11)).toBe(false)
})
```

Дублирование уходит, но работает это в рамках одного файла — в других модулях коллекцию всё равно задают заново, и это нормально.

Чрезмерное обобщение создаёт скрытые связи: изменили общую коллекцию — упали многие тесты, завязанные на её длину и значения:

```
import _ from 'lodash'

const coll = [1, 2, 3, 4]

test('filter', () => {
  // Выбираем только чётные
  expect(_.filter(coll, v => v % 2 === 0)).toEqual([2, 4])
})
```

Добавите ещё одно чётное число — тест сломается, хотя коллекцию почти наверняка придётся расширять.

Вывод: дублирование сокращать нужно, но не ценой хрупких общих фикстур на весь проект.

Не всё выносится в константы модуля — особенно динамические данные:

```
const now = Date.now() // текущий timestamp

test('first example', () => console.log(now))
test('second example', () => console.log(now))

//  console.log __tests__/index.test.js:3
//    1583871515943
//
//  console.log __tests__/index.test.js:4
//    1583871515943
```

Модуль загружается один раз — код на верхнем уровне (включая `now`) выполняется до прогона тестов. С каждым тестом «сейчас» в константе всё дальше от реального времени.

*Важно не забыть: функция test не запускает тест на выполнение. Она добавляет его внутрь Jest, а вот он уже решает, когда выполнить этот тест. Поэтому между загрузкой модуля и отработкой тестов проходит неопределённое время.*

Для этого в фреймворках есть **хуки** — функции до или после тестов. Пример: свежая дата перед каждым тестом:

```
let now

// Запускается перед каждым тестом
beforeEach(() => {
  now = Date.now() // текущий timestamp
})

test('first example', () => console.log(now))
test('second example', () => console.log(now))

//  console.log __tests__/index.test.js:9
//    1583871515943
//
//  console.log __tests__/index.test.js:10
//    1583871515950
```

`beforeEach(callback)` принимает функцию с подготовительным действием — не обязательно создание переменных (это может быть файловая система и т.д.). Чтобы данные были видны в тестах, их кладут в переменные уровня модуля — внутри колбека они останутся локальными.

Ещё пример:

```
const obj = {}

test('Add number', () => {
  add(obj, 'key', 12)
  expect(obj).toEqual({ key: 12 })
})

test('Add string', () => {
  add(obj, 'key', 'value')
  expect(obj).toEqual({ key: 'value' })
})
```

`add()` мутирует объект — второй тест упадёт. Решение:

```
let obj
beforeEach(() => {
  obj = {}
})
```

Даже разовую инициализацию лучше делать в `beforeAll(callback)` — он срабатывает один раз перед всеми тестами модуля:

```
import fs from 'fs'

let fileData

beforeAll(() => {
  fileData = fs.readFileSync('path/to/file')
})

// Такой вызов на уровне модуля (вне хуков), в общем случае, считается неправильным подходом
// fileData = fs.readFileSync('path/to/file');
```

Код на уровне модуля выполняется вне контроля Jest — фреймворк не знает, когда всё готово к запуску тестов. Подробнее об асинхронности — в следующих темах курса.
