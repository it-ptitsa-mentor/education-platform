# Разработка через тестирование

Когда писать тесты? Три распространённых подхода:

- После кода

- Вместе с кодом

- До кода

При системном тестировании (браузер, сценарии пользователя) выбора часто нет — тесты пишут после реализации.

На уровне модульных и интеграционных тестов вариантов больше. Подход «сначала код, потом тесты» — один из слабейших: пока пишете, вы и так многократно запускаете код.

В учебных задачах проверка проста:

```
// Легко запустить, легко проверить, что результат правильный
capitalize('hello') // Hello
```

В продакшене подготовка данных и проверка результата могут занимать минуты:

```
// Сложно подготовить данные. Сложно проверить результат работы.
// Загрузка товаров из 1C в базу данных
loadGoodsFrom1c()
```

Здесь выручает вариант «тесты до кода». Звучит странно, но на практике удобно.

Нужна функция, повторяющая строку N раз:

```
repeat('$', 3) // $$$
```

Вход и выход известны — тест уже можно написать:

```
test('repeat', () => {
  expect(repeat('$', 3)).toBe('$$$')
})
```

Опытный разработчик потратит на такой тест секунды; дальше достаточно `jest` в консоли.

Ещё плюс: сначала думаете об **интерфейсе** и использовании, а не о внутренней «красоте».

Подход «тесты до кода» называют **Test-Driven Development (TDD)**.

<!-- IMG (из Buildin, перезалить отдельно) -->
Классический TDD — цикл: красный тест → минимальный код → зелёный → повтор. Так наращивают приложение.

*Сейчас по инерции часто имеют в виду именно такой «мелкий» TDD с тестами на каждую функцию. Есть и другой взгляд — меньше тестов на внутренности, больше на поведение снаружи. Подробнее — в материалах ниже.*
