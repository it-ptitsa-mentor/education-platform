# Причины тестирования

Какую главную задачу должны решать тесты? От ответа зависит, как писать проверки правильно — и чего избегать.

Допустим, есть функция `capitalize(text)`, которая делает заглавной первую букву строки:

```javascript
capitalize('hello') // 'Hello'
```

Один из вариантов реализации:

```javascript
const capitalize = (text) => {
  const firstChar = text[0].toUpperCase()
  const restSubstring = text.slice(1)
  return `${firstChar}${restSubstring}`
}
```

После написания функции её проверяют: в REPL вызывают с разными аргументами:

```javascript
node
Welcome to Node.js v12.4.0.
> capitalize('hello')
'Hello'
> capitalize('how are you')
> 'How are you'
```

Так вручную убеждаются, что для выбранных входов всё работает. Заметили баг — поправили код и повторили проверку.

Это и есть тестирование — только **ручное**. Цель та же: убедиться, что поведение верное. Не важно, *как* устроена реализация внутри — это и есть главный ответ на вопрос в начале урока.

> Тесты проверяют, что код (или приложение) работает корректно. И не заботятся о том, как конкретно написан код, который они проверяют.

## Автоматические тесты

Автотесту нужно повторить те же проверки, что вы делали руками. Для начала хватит `if` и исключений.

Если с исключениями вы ещё не знакомы — в этом курсе достаточно понимать зачем они и как выглядят. В предыдущих уроках встречались «случайные» ошибки: вызов несуществующей функции, обращение к несуществующей константе. Исключение можно **создать намеренно** — это как раз нужно в тестах:

```javascript
// Дословно: выбросить новую ошибку
// Исключения бросают
throw new Error('описание исключения')
// Код, следующий за этим выражением, не выполнится, а сам скрипт завершится с ошибкой
console.log('nothing')
```

Пример теста:

```javascript
if (capitalize('hello') !== 'Hello') { // Если результат функции не равен ожидаемому значению
  // Выбрасываем исключение и завершаем выполнение теста
  throw new Error('Функция работает неверно!')
}
```

Тесты — обычный код: то же окружение, те же стандарты оформления, те же возможные ошибки в самих тестах. Писать «тесты на тесты» обычно не стоит: идеальной защиты не бывает, а бесконечное усложнение раздувает стоимость разработки. Нашли ошибку в тесте — исправили и идёте дальше ;)

Тесты обычно кладут в отдельную папку в корне проекта — чаще всего *tests*, хотя встречаются и другие имена:

```javascript
src/
├── bin
│   └── cli.js
├── half.js
└── index.js
tests/
└── half.test.js
```

Структура зависит от фреймворка. В простых случаях она зеркалит исходники: если `capitalize(text)` в *src/capitalize.js*, тест логично положить в *tests/capitalize.test.js* и импортировать функцию оттуда. Суффикс *test* в имени файла явно помечает назначение.

При любых правках функции имеет смысл снова гонять тесты:

```javascript
node tests/capitalize.test.js
# Если все хорошо, код молча выполнится.
# Если есть ошибка, то будет выведено сообщение об ошибке.
```

## Как пишутся тесты

Никакой магии: импортируете тестируемый код, вызываете с нужными аргументами, сравниваете результат с ожиданием.

Поменялась сигнатура (имя, аргументы, возвращаемое значение) — тесты, скорее всего, придётся обновить. Если снаружи всё то же, а внутри переписали реализацию:

```javascript
const capitalize = (text) => {
  const [firstChar, ...restChars] = text
  return `${firstChar.toUpperCase()}${restChars.join('')}`
}
```

— старые тесты должны проходить без правок.

Хороший тест не опирается на детали реализации — так он устойчивее к рефакторингу.

### Сколько и какие нужно писать проверки?

100% гарантии для всех возможных входов не достичь: пришлось бы перебрать бесконечное множество вариантов. Без тестов гарантий нет вовсе — остаётся вера автору кода.

Ориентир — **разнообразие входов**. У функции есть один или несколько «основных» сценариев. Для `capitalize()` — обычное слово; одной проверки на этот случай часто достаточно. Дальше смотрят на **пограничные случаи** — ситуации, где код может вести себя иначе:

- Работа с пустой строкой

- Обработка `null`

- Деление на ноль (в большинстве языков вызывает ошибку)

- Специфические ситуации для конкретных алгоритмов

Для `capitalize()` граничный случай — пустая строка:

```javascript
if (capitalize('') !== '') {
  throw new Error('Функция работает неверно!')
}
```

После такого теста видно: версия из начала урока падает — обращение к `text[0]` без проверки длины. Исправленный вариант:

```javascript
const capitalize = (text) => {
  if (text === '') {
    return ''
  }
  const firstChar = text[0].toUpperCase()
  const restSubstring = text.slice(1)
  return `${firstChar}${restSubstring}`
}
```

Во многих задачах границы требуют отдельных веток в коде — тесты должны задевать каждую такую ветку. Но ветвления могут комбинироваться: два независимых `if` дают четыре пути выполнения:

- Функция выполнилась так, что не был выполнен ни один условный блок

- Функция выполнилась так, что был выполнен только первый условный блок

- Функция выполнилась так, что был выполнен только второй условный блок

- Функция выполнилась так, что были выполнены оба условных блока

Совокупность таких путей — **цикломатическая сложность**; по ней оценивают, сколько разных сценариев стоит покрыть тестами.

Иногда «граница» не связана с `if` — например, ошибки в индексах массивов. Код может работать почти всегда и ломаться в редких случаях:

```javascript
// В этой функции забыли отнять единицу от длины
// Этот код сработает в некоторых ситуациях, когда последний элемент undefined или в массиве нет элементов
// Но в остальных случаях вернёт неверное значение
const last = elements => elements[elements.length]
```

### Проверка входных данных

Отдельная тема — неверный **тип** аргумента: что если в `capitalize()` передать число? Нужен ли такой тест?

*Ещё один интересный вопрос. Нужно ли внутри capitalize() обрабатывать такие ситуации? Ответ — не нужно. Иначе код превратится в мусорку, а пользы от этого мало. Всё равно должны быть тесты, которые проверяют, что система работает в целом, а они обычно выявляют проблемы кода на более нижних уровнях*.

Ответственность за корректные аргументы лежит на вызывающем коде. Если он хорошо протестирован, ошибка типа либо всплывёт выше, либо не возникнет.

Даже при обработке ошибок внутри функции не стоит гнаться за тестом на каждый частный случай — это раздувает поддержку. Важно вовремя остановиться и перейти к другому коду.

## Собирая всё вместе

Итоговая структура:

```javascript
src/
└── capitalize.js
tests/
└── capitalize.test.js
```

Содержимое теста:

```javascript
import capitalize from '../src/capitalize.js'

if (capitalize('hello') !== 'Hello') {
  throw new Error('Функция работает неверно!')
}

if (capitalize('') !== '') {
  throw new Error('Функция работает неверно!')
}

console.log('Все тесты пройдены!')
```

Запуск:

```javascript
node tests/capitalize.test.js
```

При успехе увидите *Все тесты пройдены!* При ошибке в тесте или в коде сработает исключение с понятным сообщением.

Ещё пример: `set()` записывает значение в свойство объекта; если ключ уже занят — ничего не меняет. По аналогии с `capitalize()` напишем тест на установку в пустой объект и на повторную запись в тот же ключ:

```javascript
import set from '../src/set.js'

const obj = {}

set(obj, 'key1', 'value1')
if (obj.key1 !== 'value1') {
  throw new Error('Функция работает неверно!')
}

set(obj, 'key1', 'next value')
if (obj.key1 === 'next value') {
  throw new Error('Функция работает неверно!')
}

console.log('Все тесты пройдены!')
```
