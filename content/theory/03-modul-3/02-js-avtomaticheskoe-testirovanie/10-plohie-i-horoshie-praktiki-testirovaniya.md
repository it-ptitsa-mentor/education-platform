# Плохие и хорошие практики тестирования

Тесты — тоже код; их можно писать хорошо и плохо. У проверок есть свои антипаттерны — разберём часть из них.

## Взаимное влияние тестов

Тесты не должны зависеть друг от друга: каждый должен проходить так, будто остальных нет.

Нарушить легко — общий файл, переменная, запись в БД. Тогда тесты падают «случайно»: отдельно зелёные, вместе — нет:

```
let user

test('first', () => {
  user = { name: 'Vasya' }
  // ...
})

test('second', () => {
  // Используется пользователь, которого создал другой тест!
  // Этот тест зависит от того, как работает предыдущий тест,
  // и не может работать без последовательного запуска обоих тестов.
  user.name = 'Petya'
})
```

Чаще всего так бывает с БД и файловой системой.

## Условные конструкции в тесте

```
test('something', () => {
  if (/* что-нибудь */) {
    // Выполняем код одним способом
    // Проверка может быть тут
  } else {
    // Выполняем код другим способом
    // Проверка может быть тут
  }
  // Проверка может быть тут
});
```

Ветвление внутри одного `test` — это несколько сценариев в одной обёртке. Так не пишут.

## Тест вне тестов

`beforeEach` готовит среду; `test` вызывает код под проверку и делает assert. Иногда наоборот:

```
let result

beforeEach(() => {
  // Вызывается тестируемый код. Это противоречит идее beforeEach.
  result = sum(5, 9)
})

test('result', () => {
  // Здесь только проверка
  expect(result).toEqual(14)
})
```

Тестируемый вызов в хуке усложняет чтение теста.

## Слишком сильная детализация

Из интернета тянут идею «один assert — один тест». Получается пять файлов с копипастой вместо одного сценария:

```
test('create user', () => {
  const user = { name: 'Mark', age: 28 }

  // Тут код, добавляющий пользователя в базу данных

  expect(user.age).toEqual(28)
})
```

```
test('create user 2', () => {
  const user = { name: 'Mark', age: 28 }

  // Тут код, добавляющий пользователя в базу данных

  expect(user.name).toEqual('Mark')
})
```

Обычно растёт только объём и цена рефакторинга.

## Глубокая вложенность

`describe` группирует тесты и задаёт свои хуки — полезно, но легко увлечься:

```
describe('', () => {
  describe('...', () => {
    describe('...', () => {
      test('should be valid', () => { /* ... */ })
    })
  })
})
```

Глубокая иерархия плохо читается и цементирует структуру.

## Код с тестами писать дольше, чем код без тестов

Хороший вопрос на собеседовании. Некоторые виды тестов действительно дороги, но **ежедневные** модульные тесты должны **ускорять** разработку:

- Тесты подталкивают к удачному дизайну API

- Подготовку данных делают один раз

- Сложную проверку результата автоматизируют матчеры

- Рефакторинг безопаснее при зелёном CI

- Меньше стресса от «а вдруг я что-то сломал»
