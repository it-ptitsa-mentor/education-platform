## Цель

Каждое задание тренажёра должно иметь честную проверку: тест **проходит на эталоне
`__solution__/`** и **падает на стартере** (статус `validated`). Источник тестов —
**оригинальные тесты Hexlet**, вытянутые из IDE-контейнеров
(`pnpm import:hexlet-files`, runbook: `docs/prompts/exercise-tests-hexlet-import.md`).
Там, где автопроверка структурно невозможна (терминал/сеть/визуал) — честный режим
«самопроверка» вместо фейкового PASS.

## ⚠️ Ключевая находка: прод проверяет код в статик-режиме

Фронт на проде собран с `VITE_STATIC_API=true` → проверка идёт браузерным
`runBrowserExerciseCheck` (слабые эвристики по kind), а **не** настоящими vitest-тестами
заданий (`__tests__/`). Поэтому «validated» в отчёте (про CI/серверные тесты) **не
равно** реальной проверке на проде — стартеры/пустые решения проходили как «выполнено».

- **Временный guard** (PR #49): нетронутый стартер на проде отклоняется
  (`isPristineStarter` в `packages/web/src/lib/pristine-starter.ts`). Закрывает самый
  грубый кейс. Заодно этим PR починен `tsc --noEmit` в `static-api.ts` (падал и на main).
- **Полное решение — инфра (2 шага):** (1) поднять Backend API на VPS, (2) пересобрать
  фронт с `VITE_STATIC_API=false` + `VITE_API_BASE_URL` на `/api`. Задача
  `6h2Xmv9M2x4CQ8fF` (за сервер отвечает не Илья). Конфиги готовы (PR #46), но не
  применены на VPS.

## Известные ограничения MVP

1. **Прогресс и черновики — в localStorage** (per-slug). Смена браузера/устройства =
   потеря прогресса. Серверный прогресс = бэкенд + аккаунты (Фаза 4), не входит в MVP.
2. **Серверная проверка кода не активна на проде** до подъёма Backend API — см. находку
   про статик-режим выше. Guard #49 прикрывает грубые кейсы.
